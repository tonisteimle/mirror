Mirror

What is Mirror

Mirror is a description language for user interfaces. One syntax for structure, styling, and interactions – that compiles to React.

Building UI today means juggling three languages: HTML for structure, CSS for styling, JavaScript for behavior. Each lives in different files, follows different rules, and requires constant mental context-switching. A simple button with hover states and click handlers spans dozens of lines across multiple files.

Mirror takes a different approach. It's a single, declarative language where components, their appearance, and their behavior live together. You describe what you want, and Mirror handles the rest.

Mirror is for anyone who wants to build interactive prototypes fast. Full control over layout, styling, and behavior – in a fraction of the code.

Tokens

Components centralize structure. But what about values – colors, sizes, spacing? Tokens let you define them once and reuse everywhere.

Imagine changing your brand color from blue to purple. Without tokens, you'd search through every component, find every #2271c1, and replace it manually. With tokens, you change one line – and the entire UI updates. Tokens are the foundation of a design system: named values that give meaning to raw numbers and hex codes.

A token is a named value that can be used throughout the design. Tokens start with $.

Change $primary-color once, and every element using it updates. The suffix in the name (-color, -radius, -padding) tells Mirror which property it represents.

Tokens vs. Definitions

Tokens hold single values – one color, one number, one string. But what if you want to reuse a combination of properties, like a text style with font, size, and weight together?

That's what Component Definitions are for. While tokens store individual values, definitions bundle multiple properties into reusable components:

Think of it this way: Tokens are your design system's raw materials (colors, spacing, sizes). Definitions are the recipes that combine those materials into consistent patterns (text styles, button variants, card layouts).

Semantic Token Hierarchy

Design systems often organize tokens in layers. Base tokens define raw values. Semantic tokens assign meaning. Component tokens connect to specific UI elements.

This structure allows changes at any level. Swap $blue-500 for a different shade, and all primary elements update. Reassign $primary-color to a different base color, and the meaning stays intact. Change $button-color independently if buttons need a unique treatment.

Interactivity

Everything so far has been static – components that look good but don't respond to users. Now we'll bring them to life.

Interactivity in Mirror follows one simple principle that explains everything:

EVENT → ACTION → STATE

Something happens → Mirror does something → The UI changes

That's the entire mental model. A user clicks a button (event), Mirror toggles it (action), and the button switches to its "on" state (state). Once you understand this flow, everything else is just variations and combinations.

Variables & State

You've built static interfaces – components that always look the same. Now let's make them come alive. A button that counts clicks. A toggle that remembers its state. A display that updates in real-time.

The secret is variables: values that can change. When a variable changes, every part of your UI that uses it updates automatically. No manual refresh, no complex wiring – just declare what depends on what, and Mirror handles the rest.

Conditional Display

Your UI responds to clicks and updates values. But what if you want to show completely different content based on state? A login button that becomes a logout button. An error message that only appears when needed. A theme that switches between light and dark.

Conditions let you control what appears, not just what it says. Show this component or that one. Apply this style or that one. Mirror evaluates your conditions and renders the right UI automatically.

Data Tab

For complex prototypes with structured data, the Data tab provides a cleaner way to define schemas and instances separately from your layout code.

Instead of embedding JSON arrays in your Layout tab, you define data types and instances using a simple, readable syntax. This separates concerns: Layout defines how data looks, Data defines what the data is.

Defining Schemas

A schema defines the structure of your data. Use a colon after the type name, then indent fields with their types: 

// In the Data tab:
Task:
  title: text
  done: boolean
  priority: number

Category:
  name: text
  color: text

Each field has a name and a type. Available types are:

Field Types

textString values like "Buy groceries"
numberNumeric values like 42 or 3.14
booleantrue or false
TypeNameReference to another schema (e.g., Category)

Creating Instances

After defining schemas, create instances using - prefix. Values are listed in field order: 

// Schema
Category:
  name: text
  color: text

// Instances (values in field order, comma-separated)
- Category "Arbeit", "#3B82F6"
- Category "Privat", "#10B981"
- Category "Dringend", "#EF4444"

For schemas with many fields: 

Task:
  title: text
  done: boolean
  priority: number

// Instances: title, done, priority (comma-separated)
- Task "Einkaufen", false, 1
- Task "Sport machen", true, 2
- Task "Meeting vorbereiten", false, 3

References Between Types

Use index notation to reference other instances. Category[0] refers to the first Category instance: 

Category:
  name: text

Task:
  title: text
  done: boolean
  category: Category    // Reference to Category type

// Categories
- Category "Arbeit"
- Category "Privat"

// Tasks referencing categories by index (comma-separated)
- Task "Email schreiben", false, Category[0]    // References "Arbeit"
- Task "Joggen gehen", true, Category[1]        // References "Privat"
- Task "Report erstellen", false, Category[0]   // References "Arbeit"

The index [0] refers to the first instance of that type, [1] to the second, and so on.

Using Data in Layout

Use the data property to bind a container to your data. The container repeats its children for each record: 

// In Data tab:
Task:
  title: text
  done: boolean

- Task "Buy groceries", false
- Task "Call mom", true

// In Layout tab:
TaskList data Tasks ver gap 8
  Card padding 12 radius 8
    // Each Card repeats for each Task record

Data Binding Syntax

data TasksIterate over all Task records
data Tasks where done == falseFilter: only incomplete tasks
data Tasks where priority > 2Filter with numeric comparison

Note: Field binding syntax (Tasks.title) to display data values directly is planned for a future release.

Complete Example

Here's a full example with multiple related types: 

// ===== DATA TAB =====

// Schema definitions
Category:
  name: text
  icon: text
  color: text

Task:
  title: text
  done: boolean
  category: Category

// Category instances
- Category "Work", "briefcase", "#3B82F6"
- Category "Personal", "user", "#10B981"
- Category "Urgent", "alert-circle", "#EF4444"

// Task instances
- Task "Prepare presentation", false, Category[0]
- Task "Buy birthday gift", false, Category[1]
- Task "Fix critical bug", false, Category[2]
- Task "Morning workout", true, Category[1]
- Task "Team meeting", true, Category[0]
// ===== LAYOUT TAB =====

// The container repeats for each Task record
TaskList data Tasks ver gap 8 padding 16
  Card hor between vertical-center bg #1A1A23 padding 12 radius 8
    Left hor gap 12 vertical-center
      Icon icon "file-text"
      Title "Task item"
    Status
      Icon icon "circle" color #333

// With filter: only show incomplete tasks
TodoList data Tasks where done == false ver gap 8
  Card background #1A1A23 padding 12 radius 8
    Text "Todo item"

AI-Powered Data Generation

When you've defined schemas in the Data tab, a Generate button appears. Click it to have AI create realistic sample data based on your schema definitions.

The AI understands your field names and types, generating contextually appropriate values:

  • title: text → Generates meaningful titles
  • email: text → Generates realistic email addresses
  • price: number → Generates appropriate price values
  • done: boolean → Generates true/false mix

This is especially useful for prototyping: define your data structure once, then generate 10, 20, or 50 realistic entries instantly.
