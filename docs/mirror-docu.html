Mirror

What is Mirror

Mirror is a description language for user interfaces. One syntax for structure, styling, and interactions – that compiles to React.

Building UI today means juggling three languages: HTML for structure, CSS for styling, JavaScript for behavior. Each lives in different files, follows different rules, and requires constant mental context-switching. A simple button with hover states and click handlers spans dozens of lines across multiple files.

Mirror takes a different approach. It's a single, declarative language where components, their appearance, and their behavior live together. You describe what you want, and Mirror handles the rest.

Mirror is for anyone who wants to build interactive prototypes fast. Full control over layout, styling, and behavior – in a fraction of the code.

Tokens

Components centralize structure. But what about values – colors, sizes, spacing? Tokens let you define them once and reuse everywhere.

Imagine changing your brand color from blue to purple. Without tokens, you'd search through every component, find every #2271c1, and replace it manually. With tokens, you change one line – and the entire UI updates. Tokens are the foundation of a design system: named values that give meaning to raw numbers and hex codes.

A token is a named value that can be used throughout the design. Tokens start with $.

Change $primary-col once, and every element using it updates. The suffix in the name (-col, -rad, -pad) tells Mirror which property it represents.

Tokens vs. Definitions

Tokens hold single values – one color, one number, one string. But what if you want to reuse a combination of properties, like a text style with font, size, and weight together?

That's what Component Definitions are for. While tokens store individual values, definitions bundle multiple properties into reusable components:

Think of it this way: Tokens are your design system's raw materials (colors, spacing, sizes). Definitions are the recipes that combine those materials into consistent patterns (text styles, button variants, card layouts).

Semantic Token Hierarchy

Design systems often organize tokens in layers. Base tokens define raw values. Semantic tokens assign meaning. Component tokens connect to specific UI elements.

This structure allows changes at any level. Swap $blue-500 for a different shade, and all primary elements update. Reassign $primary-col to a different base color, and the meaning stays intact. Change $button-col independently if buttons need a unique treatment.

Interactivity

Everything so far has been static – components, styling, forms. Now let's make things respond to user actions.

Traditional UI frameworks separate event handling into JavaScript callbacks, Redux actions, or hook functions. Mirror takes a different approach: behavior belongs where you see it. Events and actions are written directly on the component, making the connection between UI and behavior immediately visible.

Events

Events are written directly on the component. An event triggers an action.

onclick is the event. toggle is the action. The pattern is always: Component event action.

States

Components can have multiple visual states. Each state defines different styling that applies when the state is active.

Events trigger actions, but where does the visual change happen? In Mirror, components can define states – named configurations of properties. When a state becomes active, its properties override the defaults. This separates "what triggers a change" (events) from "what the change looks like" (states).

In traditional development, you might toggle CSS classes with JavaScript, or manage state variables that control styling through conditionals. Both approaches scatter the visual definition across multiple places. With Mirror states, all visual variations live together on the component itself. You see at a glance what a button looks like when inactive, active, hovered, or disabled – without jumping between files or mental context-switching.

Behaviors

Centralized events separate structure from behavior. Behaviors take this further – they define reusable interaction patterns for complex UI components.

A dropdown needs click-outside-to-close, keyboard navigation, item highlighting, and selection. In traditional development, this means ~350 lines of JavaScript with event listeners, state management, and cleanup code. Mirror behaviors reduce this to a few declarative lines that read like prose.

Behavior States

We've seen system states like hover and focus that are automatic. Behavior states are activated by actions – they give you full control over when states change.

highlight self activates the highlighted state on hover. The state defines the visual change.

Common Behavior States

  • highlighted – visual focus (keyboard/mouse navigation)
  • selected – persistent selection
  • active / inactive – tab or toggle states
  • expanded / collapsed – accordion states

The Event-Action-Target Pattern

Behaviors follow a consistent pattern that reads like a sentence: 

on-event [key] action target

Examples:

  • onclick close Dropdown – "On click, close the Dropdown"
  • onkeydown escape close self – "On Escape key, close myself"
  • onhover highlight self – "On hover, highlight myself"
  • onkeydown arrow-down highlight next – "On arrow-down, highlight the next item"

Each library component has:

  • Slots – Named positions for your content
  • States – Visual states that change automatically
  • Behavior – Built-in interactions (clicks, keyboard, focus)

Tabs

Organize content into switchable panels. Only one panel is visible at a time. Keyboard navigation (arrow keys) is built-in.

Tabs Slots

  • Tabs – Container for tab buttons
  • Tab – Individual tab button (multiple)
  • TabContent – Content for each tab (multiple, matched by order)

Tab States

  • inactive – Tab is not selected (default)
  • active – Tab is currently selected

Styling Tab States

Create tabs for "Code", "Preview", and "Output". Style the active tab with a colored bottom border.

Accordion

Collapsible sections where only one (or multiple) can be open at a time. Great for FAQs, settings panels, or navigation menus.

Accordion Slots

  • AccordionItem – A single collapsible section (multiple)
  • Trigger – The clickable header
  • Content – The collapsible content

AccordionItem States

  • collapsed – Content is hidden (default)
  • expanded – Content is visible

Dropdown

A menu that appears below its trigger on click. Closes automatically when clicking outside or selecting an item.

Dropdown Slots

  • Trigger – Button that opens the menu
  • Content – The dropdown menu container
  • Item – A menu item (multiple)
  • Separator – A dividing line (optional)

States

  • closed – Menu hidden (default)
  • open – Menu visible

Select

A styled dropdown for selecting a single value. Replaces native <select> with full styling control.

Select Slots

  • Trigger – Shows current selection, opens on click
  • Options – The options container
  • Option – An option (requires value property, multiple)
  • Group – Optional group container for options
  • Label – Optional label for a group

Option States

  • unselected – Not the current value
  • selected – Currently selected item

Switch Slots

States

Slider

A range input for selecting a value within a range.

Slider Slots

  • Range – The filled portion (0 to current value)
  • Thumb – The draggable handle

  • value – Current value
  • step – Increment step

  • expanded – Content visible

ContextMenu

A menu that appears on right-click. Useful for providing context-specific actions.

ContextMenu Slots

  • Content – The menu container
  • Item – A menu item (multiple)
  • Separator – Dividing line (optional)

Quick Reference

All properties, keywords, and patterns at a glance.

Layout

  • hor – hor (flex row)
  • ver – ver (flex column)
  • gap 16 – space between children
  • between – space-between
  • wrap – flex wrap
  • grow / fill – flex grow
  • cen – cen both axes
  • hor-l / hor-cen / hor-r – hor alignment
  • ver-t / ver-cen / ver-b – ver alignment
  • grid 3 – 3 equal columns
  • grid 20% 80% – percentage columns
  • grid auto 250 – auto-fill with min width

Sizing

  • w 200 / h 100 – width/height in pixels
  • w full / h full – 100%
  • full – both 100%
  • minw / maxw / minh / maxh – constraints

Spacing

  • pad 12 – pad all sides
  • pad 8 16 – ver horizontal
  • pad l 12 – left only (also: r, u, d, l-r, u-d)
  • mar 12 – margin (same syntax)

Colors

  • col #2271c1 – unified color (background for containers, text for text)
  • boc #333 – border color
  • textCol #fff – override auto text color

Border

  • rad 8 – border radius
  • bor 1 – border width (solid)
  • bor 1 #333 – border with color (shorthand)
  • bor l 2 dashed #333 – directional compound border

Typography

  • size 18 – font size
  • weight 600 / weight bold – font weight
  • font "Inter" – font family
  • line 1.5 – line height
  • align center – text alignment
  • italic / underline / uppercase / lowercase
  • truncate – ellipsis overflow

Visual

  • opacity 0.5 / opa 0.5 – transparency
  • shadow sm / md / lg – box shadow
  • z 10 – z-index
  • cursor pointer – cursor style
  • hidden / visible – visibility
  • icon chevron-down – Lucide icon

Scroll

  • scroll / scroll-ver – ver scroll
  • scroll-hor – hor scroll
  • scroll-both – both directions
  • snap – scroll snap
  • clip – overflow hidden

Hover States

  • hover-col – background on hover
  • hover-boc / hover-bor / hover-rad
  • hover-opacity / hover-scale

Events & Actions

  • onclick / onchange / oninput / onfocus / onblur
  • onclick-outside – click outside element
  • onkeydown escape / enter / arrow-down / arrow-up – key events
  • toggle – switch state
  • open Name / close – overlays
  • show Name / hide Name – visibility
  • highlight self / next / prev – navigation highlight
  • select self / highlighted – selection
  • filter Container – filter items by input
  • page Name – navigation
  • assign $var to value – set variable

Logic

  • if $condition ... else – conditional rendering
  • if $x then prop value else prop value – conditional property
  • each $item in $list – iteration
  • and / or / not – logical operators
  • == / != / > / < / >= / <= – comparison

Animations

  • Types: fade, scale, slide-up/down/left/right, spin, pulse, bounce
