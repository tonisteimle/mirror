Abschnitt 1

Syntax Grundlagen

Zeilenstruktur

ComponentName property value property value "text content"

Einrückung

Kinder werden mit 2 Spaces eingerückt:

Parent
  Child
    Grandchild

Kommentare

// Dies ist ein Kommentar
Button "Click"  // Inline-Kommentar

Strings

Doppelte Anführungszeichen für Textinhalt:

Button "Click me"
Label "Hello World"

Zahlen

Zahlen sind standardmäßig Pixel. Verwende % für Prozent:

w 200       // 200px
w 50%       // 50%
h 100       // 100px
pad 16      // 16px

Dimension Shorthand

Die ersten zwei Zahlen nach dem Komponenten-Namen werden automatisch als Breite und Höhe interpretiert:

// Shorthand
Box 300 400 pad 16           // w 300, h 400
Card 200 pad 8               // nur w 200

// Äquivalent zu
Box w 300 h 400 pad 16
Card w 200 pad 8

// Mischen möglich
Panel 300 h 400              // w implizit, h explizit

Konsistent mit Image: Bei Image wird der String als src interpretiert, gefolgt von Breite und Höhe: Image "foto.jpg" 800 600

Farben

Hex-Notation, optional mit Alpha:

col #3B82F6        // RGB
col #3B82F680      // RGBA (50% Transparenz)
col #FFF          // Kurzform

Komponenten-Namen

Button "Click"    // Großbuchstabe → Komponente
header hor        // Kleinbuchstabe → Element

Farben (Unified Color System)

PropertyBeschreibungBeispiel
colKomponenten-Farbe (kontextabhängig)col #3B82F6
bocRahmenfarbeboc #333333

Unified Color: col ist die einzige Farb-Property. Bei Container-Komponenten (Box, Button, Card) wird col zur Hintergrundfarbe. Bei Text-Komponenten (Text, Label, Title) wird col zur Textfarbe.

// Container: col → backgroundColor
Button col #3B82F6 "Click"  // Blauer Hintergrund
Card col #1E1E1E            // Dunkler Hintergrund

// Text: col → color
Text col #FFFFFF "Hello"   // Weiße Schrift
Label col #888888          // Graue Schrift

// Container mit Text: Sugar Syntax
Button col #3B82F6 "Click" col #FFFFFF
// ↑ Erste col → Button-Hintergrund, zweite col → Text-Farbe

Rahmen (Compound Properties)

PropertyBeschreibungBeispiel

Compound Syntax: bor kann mehrere Werte in beliebiger Reihenfolge annehmen: Richtung (l r u d l-r u-d), Breite (Zahl), Stil (solid dashed dotted), Farbe (#hex). Der Parser erkennt die Typen automatisch.

// Einfacher Rahmen
Card bor 1                        // 1px solid
Card bor 1 #333                   // 1px solid #333
Card bor 2 dashed #3B82F6        // 2px dashed #3B82F6

// Mit Richtung
Sidebar bor l 1 #333             // Nur links
Header bor d 1 #333              // Nur unten
Card bor l-r 2 dashed #3B82F6   // Links und rechts

// Flexible Reihenfolge (alle äquivalent)
Box bor 1 solid #333
Box bor #333 1 solid
Box bor solid #333 1

// Eckenradius
Card rad 12                       // Alle Ecken
Tag rad 99                        // Pill-Form

Flacher Zugriff

Greife auf verschachtelte Elemente direkt per Namen zu, unabhängig von der Hierarchie:

// Definition mit Verschachtelung
Header: hor between ver-cen h 60 pad l-r 24
  Left: hor gap 16 ver-cen
    Logo: w 120 h 32
    Nav: hor gap 8
      Link: col #888
  Right: hor gap 12
    Avatar: w 36 h 36 rad 18

// Verwendung - direkt auf Logo zugreifen (egal wie tief)
Header
  Logo col #FF0000              // Modifiziert Logo in Left
  Avatar col #3B82F6            // Modifiziert Avatar in Right

Wichtig: Namen müssen innerhalb einer Komponente eindeutig sein. Bei Duplikaten gibt der Parser einen Fehler aus.

List Items (Neue Instanzen)

Verwende - um neue Instanzen eines Elements zu erstellen:

// Definition
Dropdown: ver w 200
  Trigger: pad 12 col #2A2A3E
  Menu: ver
    Item: pad 8 12

// Verwendung - füge mehrere Items hinzu
Dropdown
  Trigger "Optionen"
  - Item "Alle anzeigen"
  - Item "Nur aktive"
  - Item "Archivierte"

Das - funktioniert wie bei YAML-Listen: Jedes - erstellt eine neue Instanz.

Kombination: Flacher Zugriff + List Items
Card: ver pad 16 gap 12
  Title: size 18 weight 600
  Tags: hor gap 8 wrap
    Tag: pad 4 8 rad 4 col #3B82F6 size 12

// Verwendung
Card
  Title "Projekt Alpha"        // Flach: modifiziert Title
  - Tag "React"                // Neu: fügt Tag hinzu
  - Tag "TypeScript"           // Neu: weiterer Tag
  - Tag "UI" col #10B981        // Neu: Tag mit anderem Hintergrund

Komponenten-Properties

Jede Komponente hat intrinsische Properties. Zugriff mit Dot-Notation:

Property lesen

if Email.value                    // Truthy-Check
if Email.value == "test@test.com"  // Vergleich
if Checkbox.checked               // Boolean
if Panel.visible                  // Sichtbarkeit

Property schreiben

Submit.label = "Sending..."
Submit.disabled = true
Panel.visible = false
Error.text = "Fehler!"

Komponenten SIND der State. Kein separates State-Management nötig. Email.value statt $formData.email.

Intrinsische Properties

KomponenteProperties
Alle.visible, .disabled, .opacity, .col
Input / Textarea.value, .placeholder, .focus
Button.label, .loading
Checkbox / Switch.checked
Dialog / Overlay.open, .close
Text / Label.text
Image.src, .alt

Zentrale Events

Trenne Struktur von Verhalten mit dem events Block:

// === KOMPONENTEN ===
Input Email: placeholder "Email"
Input Password: placeholder "Password"
Text Error: col #EF4444 visible false
Button Submit: col #3B82F6 "Login"

// === LAYOUT ===
LoginForm: ver gap 16
  Input Email
  Input Password
  Text Error
  Button Submit

// === VERHALTEN ===
events
  Email onchange
    Error.visible = false

  Submit onclick
    if Email.value and Password.value
      Submit.label = "Sending..."
      page Dashboard
    else
      Error.text = "Bitte alle Felder ausfüllen"
      Error.visible = true

Inline vs. Zentral: Einfache Events inline (onclick toggle), komplexe Logik im events Block.
