Mirror

Design and code are converging. AI accelerates this shift. But the current ecosystem wasn't built for designers.

Three languages to learn. HTML for structure, CSS for styling, JavaScript for behavior. Add React, Vue, or Svelte for anything beyond static pages. Each with its own syntax, mental model, and edge cases.

Writing code takes too long. Reading AI-generated code takes even longer. The roundtrip between intent and implementation is where momentum dies.

Mirror closes this gap. One language. Structure, style, and behavior in a single line. Design tokens built in – $primary, $spacing-md – so design systems translate directly into code. Compiles to production-ready React, Svelte, or Vue.

The editor shows a live preview as you type. No build step, no refresh. Every keystroke updates the result. You see what you're building while you build it.

The goal: prototype at the speed of thought. A designer who knows Mirror isn't slower than in Figma – but the output runs.

Everything is a Component

The fundamental principle: Every element is a component. There is no difference between "primitive elements" and "components". Components can be used inline or explicitly defined.

Inline – use directly
Button pad 12 col #3B82F6 "Click me"
Explicit definition
Button: pad 12 col #3B82F6 rad 8

The colon indicates: This line doesn't render UI. It only defines how Button should look.

Reusable after definition
Button: pad 12 col #3B82F6 rad 8

Button "Save"
Button "Cancel"
Button "Delete"

UI elements are always reusable – this automatically leads to consistency. There's no distinction between "components" and other code. We always think in components, so we write that way too.

Indentation = Structure

Hierarchy is created through indentation. No tags, no brackets, no semicolons. 2 spaces = one level deeper. 

App
  Header
    Logo
    Nav
  Main
    Sidebar
    Content
      Article
  Footer
Mirror
Card
  Title "Hello"
  Body
    Text "World"
Equivalent in HTML
<Card>
  <Title>Hello</Title>
  <Body>
    <Text>World</Text>
  </Body>
</Card>

The visual structure matches the logical one – you see the hierarchy directly. No closing tags, no brackets means fewer syntax errors and roughly half as many characters as HTML.

Type Inference

Values are automatically interpreted. No type annotations, no unit suffixes. Mirror understands what you mean. 

// Numbers → Pixels
pad 12          // → padding: 12px
w 200          // → width: 200px
gap 16         // → gap: 16px

// Hex → Colors (Unified Color System)
col #3B82F6     // → Container: backgroundColor, Text: color
col #1A1A1A     // → explicit backgroundColor
boc #33333380  // → border-color with alpha

// Strings → Text
"Hello World"  // → text content
font "Inter"   // → font-family: "Inter"

// Keywords → Boolean
hor            // → flex-direction: row
cen            // → centered
italic         // → font-style: italic

// Special values
w full         // → width: 100%
h full         // → height: 100%

You write pad 12 instead of padding: "12px". No forgotten units, no quotes. Context determines meaning – Mirror understands what you mean.

Auto Text Color

When you set a background color on a container, Mirror automatically calculates the optimal text color for readability. 

// Dark background → white text (automatic)
Button col #3B82F6 "Click me"   // → text: #fff
Card col #1F2937                // → text: #fff

// Light background → dark text (automatic)
Button col #F3F4F6 "Click me"   // → text: #000
Card col #FFFFFF                // → text: #000

// Override with textCol if needed
Button col #3B82F6 textCol #FFD700 "Gold Text"

The Rule

Mirror uses luminance to determine contrast:

  • Dark backgrounds (luminance ≤ 0.5) → white text (#fff)
  • Light backgrounds (luminance > 0.5) → black text (#000)

This threshold ensures colored buttons (blue, red, green) get white text, while light grays and yellows get dark text.

Use textCol to override the automatic color.

No more guessing which text color looks good on which background. Mirror ensures readable contrast automatically – one less thing to think about.

Overriding Properties

Any property can be overridden when using a component. Just add the new property – done.

Base component
Button: pad 12 col #3B82F6 rad 8
Override on usage
// Different color
Button col #EF4444 "Delete"

// Larger
Button pad 20 size 18 "Large Button"

// Ghost style
Button col transparent bor 1 boc #3B82F6 "Ghost"

Every component is customizable – there's no closed props API. One-off adjustments don't need a predefined variant. The rule is simple: later properties win.

Inheritance with from

Create variants of a component without repeating everything. The new component inherits all properties and only overrides what's needed. 

// Base
Button: pad 12 col #3B82F6 rad 8

// Variants
DangerButton from Button: col #EF4444
GhostButton from Button: col transparent bor 1 boc #3B82F6
SmallButton from Button: pad 8 size 12
RoundButton from Button: rad 999
Multiple inheritance
SmallDangerButton from DangerButton: pad 8 size 12

Define the base once, derive variants from it. Changes to the base propagate automatically. Names like DangerButton communicate meaning, not just styling. Override for one-offs, inheritance for real variants.

Implicit Children

Strings in component definitions automatically become text children. On usage, the text is simply replaced.

Definition with text child
// "Label" becomes an implicit text child
Button: pad 8 16 col #E72A69 rad 8 "Label" #fff
Usage – override text
// "Strawberries" replaces "Label"
Button "Strawberries"
Button "Cherries"
Button "Bananas"
With styling after text
// Properties after the string belong to the text
Card: pad 16 col #1A1A23
  "Title" #fff size 18 weight bold

// Usage
Card "My Project"   // → white text, 18px, bold

You define the structure once. The text is variable, the styling stays. No passing props, no template syntax – just provide the new text.

References with $

Everything starting with $ is a reference. A unified system for all kinds of values.

Design Tokens
// Definition
$primary: #3B82F6
$spacing: 16

// Usage
Button col $primary pad $spacing
Automatic Inference
// Color tokens automatically become col
$primary: #3B82F6
Box $primary           // → col: #3B82F6

// Suffix-based inference (3 characters)
$card-pad: 24
$card-rad: 12
$border-col: #333

Card $card-pad $card-rad   // → pad: 24, rad: 12
Box $border-col            // → col: #333

Token Suffixes

  • -pad → padding
  • -mar → margin
  • -rad → border-radius
  • -gap → gap
  • -col → color
  • -boc → border-color
  • -bor → border-width

Tokens communicate what they are through their name. $card-pad is obviously a padding – why write pad again? Color values are automatically recognized. Less redundancy, more readability.

Component Properties
Card: rad 16 pad 20

// Reference to Card's radius
Button: rad Card.rad  // → 16
State Variables
// Display variable
Counter $count

// Change variable
Button onclick assign $count to $count + 1
Event Objects
Input onchange assign $name to $event.value
Checkbox onchange assign $checked to $event.checked
Iterator Items
each $task in $tasks
  Card
    Title $task.title
    Status $task.status

The $ always signals a reference – whether it's a token, variable, event data, or iterator item. Everything works the same, no imports needed, everything automatically available.

States – Visual States

Components can have multiple visual states. Each state defines how the component looks in that state.

Tab component

All states of a component are visible in one place. Named states instead of dozens of if-statements. States are a design concept – that's how designers already think.

Events & Actions

Interactivity through event handlers. An event triggers an action.

Events
onclick    // Click
onhover    // Hover
onchange   // Value changed
oninput    // Input
onfocus    // Focus gained
onblur     // Focus lost
Actions
toggle                    // Toggle state
change self to active   // To state
open Dialog              // Open overlay
open Dialog slide-up    // With animation
open Dialog fade 300   // With animation + duration
close                     // Top overlay
close Dialog             // Specific overlay
page Dashboard           // Navigation
assign $x to 1            // Set value
Animations for open/close
slide-up slide-down slide-left slide-right fade scale none
Examples
Button onclick toggle "Dark Mode"
Button onclick open SettingsDialog slide-up 300 "Settings"
Button onclick close SettingsDialog fade 150 "Close"
Tab onclick change self to active "Tab 1"
NavItem onclick page Dashboard "Dashboard"
Button onclick assign $count to $count + 1 "+"

You say what should happen, not how. open Dialog slide-up – done. No callbacks, no handler definitions. Actions cover most cases.

Named Instances

Give components a name to reference them later. Like in Visual Basic: The name is the identity.

Syntax
// Type + Name = Named Instance
Input Email: placeholder "Email" w full
Input Password: placeholder "Password" type password
Button Submit: pad 12 col #3B82F6 "Login"
Usage
LoginForm: ver gap 16
  Input Email
  Input Password
  Button Submit

The name makes the component referenceable. Email, Password, Submit – you can address them later.

Component Properties

Every component has intrinsic properties. Read and write with dot notation.

Read property
if Email.value                    // Truthy check
if Email.value == "test@test.com"  // Comparison
if Checkbox.checked               // Boolean
if Panel.visible                  // Visibility
Write property
Submit.label = "Sending..."
Submit.disabled = true
Panel.visible = false
Error.text = "Please fill in"
Image.src = "new-photo.jpg"

Components ARE the state. No separate state management. No Redux. No Vuex. UI components have properties – that's the state.

Intrinsic Properties
// All components
.visible     // Visible?
.disabled    // Disabled?
.opacity     // Transparency
.col         // Color (background for containers)

// Input / Textarea
.value       // Current value
.placeholder // Placeholder
.focus       // Set focus
// Button
.label       // Button text
.loading     // Loading state

// Checkbox / Switch
.checked     // Activated?

// Dialog / Overlay
.open        // Open?
.close       // Close

// Text / Label
.text        // Text content

Like in Visual Basic: txtEmail.Text instead of formState.email. The component is its own state.

Centralized Events

Separate structure from behavior. Define all events in one place.

The events block
events
  Email onchange
    Error.visible = false

  Submit onclick
    if Email.value and Password.value
      Submit.label = "Sending..."
      Submit.disabled = true
      page Dashboard
    else
      Error.text = "Please fill in all fields"
      Error.visible = true
Complete example
// === COMPONENTS ===
Input Email: placeholder "Email" w full
Input Password: placeholder "Password" type password
Text Error: col #EF4444 visible false
Button Submit: col #3B82F6 "Login"

// === LAYOUT ===
LoginForm: ver gap 16 pad 24
  Input Email
  Input Password
  Text Error
  Button Submit

// === BEHAVIOR ===
events
  Submit onclick
    if Email.value and Password.value
      page Dashboard
    else
      Error.visible = true

Inline vs. Centralized

// Inline – for simple cases
Button onclick toggle "Dark Mode"

// Centralized – for complex logic
events
  Submit onclick
    // Validation, API calls, navigation...

Structure and behavior separated. All interactions in one place. Readable like prose: "When Submit is clicked..."

Conditional Logic

Show components based on conditions. Change properties based on state.

Conditional components
if $isLoggedIn
  Avatar
  Button "Logout"
else
  Button "Login"
Conditional properties
// Simple
Button if $isActive then col #3B82F6 else col #333

// Multiple properties
Card if $isSelected then col #3B82F6 bor 2 else col #1A1A1A bor 0

// With comparison
Badge if $count > 0 then opacity 1 else opacity 0

The syntax reads like a sentence: if $active then col blue. No cryptic ternary operators. When the value changes, the UI updates automatically.

Iterators

Render components for each element in a list. Access properties of the current element. 

each $task in $tasks
  TaskCard
    Title $task.title
    Status $task.status
    DueDate $task.due
Nested iteration
each $category in $categories
  Section
    Title $category.name
    each $item in $category.items
      Item $item.name

Natural language: each $task in $tasks. The UI automatically adapts to the data. The variable only exists within the block – no name conflicts possible.

Slots – Named Children

Components can define named positions for children. On usage, these slots are filled.

Definition with slots
Card: ver pad 24 gap 16 col #1E1E2E rad 12
  Header: hor between ver-cen
  Title: size 20 weight 600
  Body: size 14 col #9CA3AF
  Actions: hor gap 8 hor-r
Usage – filling slots
Card
  Header
    Badge "New"
    Icon "more-vertical"
  Title "Project Name"
  Body "Project description..."
  Actions
    Button "Cancel"
    Button col $primary "Save"

The definition specifies where content can appear. The usage determines what goes there. Children are placed directly, not passed as props – like Card.Title or Card.Body in modern UI libraries.

Flat Access

Access nested elements directly by name. No matter how deep they are in the hierarchy.

Nested definition
Header: hor between h 60
  Left: hor gap 16
    Logo: w 100
    Nav: hor gap 8
  Right: hor
    Avatar: w 36 rad 18
Usage – direct access
Header
  Logo col #FF0000       // Finds Logo in Left
  Avatar col #3B82F6     // Finds Avatar in Right

You don't need to repeat the hierarchy. Names are flat, structure is preserved. This makes components easy to use without knowing their internals. Names must be unique – duplicates cause an error.

List Items

Create new instances with -. Like lists in YAML.

Definition with template
Menu: ver w 200 col #1E1E2E rad 8
  Item: pad 8 12 col #E4E4E7
Usage – multiple instances
Menu
  - Item "Profile"
  - Item "Settings"
  - Item "Logout" col #EF4444
Combined with flat access
Card: ver pad 16 gap 12
  Title: size 18
  Tags: hor gap 8
    Tag: pad 4 8 rad 4 col #3B82F6

Card
  Title "Project"         // Modifies
  - Tag "React"           // New
  - Tag "TypeScript"      // New

Without - you modify an existing element. With - you create a new instance. Simple distinction, powerful flexibility. Any element can be extended this way.

Data Binding

Variables are reactive. When a value changes, the UI updates. Use assign to change values.

Display variables
Label $username
Counter $count
Price $price " €"
Change values with assign
// Simple set
Button onclick assign $count to 0 "Reset"

// With expression
Button onclick assign $count to $count + 1 "+"
Button onclick assign $count to $count - 1 "-"

// From input
Input onchange assign $name to $event.value
Expressions
// Arithmetic
Total $price * $quantity
Discount $total * 0.1

// In conditions
if $count >= 10
  Badge "Maximum reached"

Two ways to set values:
assign $variable to value – for variables (with $ prefix)
Component.property = value – for component properties (in events block)

When a value changes, the UI updates – automatically. No setState, no dispatches: just assign $count to $count + 1. Calculations like $price * $quantity are written directly in the UI.

Content at the End

Text content always goes at the end of the line. First component, then properties, then content. 

// Structure
Component property value property value "Content"

// Examples
Button "Click"
Button col #3B82F6 "Click"
Button pad 12 col #3B82F6 rad 8 "Click"

// Dynamic content
Label $username
Greeting "Hello, " $name
Price $amount " €"

Every line follows the same pattern and is therefore quickly readable. Structure on the left, content on the right. Always the same order: what it is, how it looks, what's inside.

Scope

Variables have different visibility. From global to block-local.

Global – Tokens
// Available everywhere
$primary: #3B82F6
$spacing: 16

// Usable in any component
Button col $primary
Card pad $spacing
Global – State Variables
// Reactive variables, visible everywhere
Counter $count
Button onclick assign $count to $count + 1

// Other component can read the same variable
Badge if $count > 0 then $count
Block Scope – Iterator
each $task in $tasks
  // $task only visible here
  Card $task.title

// $task is no longer available here
Footer
Handler Scope – Event
Input onchange assign $name to $event.value
       // $event only visible in handler

You always know where a variable is valid. $task in one iterator doesn't interfere with $task in another. Design tokens are global, iterator items are local – exactly as expected.

HTML Primitives

Certain elements require native HTML tags: Image, Input, Textarea, Link, Button, H1-H6. These render as their corresponding HTML elements.

Syntax

Keyword Name? "Value" Properties

The name is optional and makes the component reusable.

Image

The string is interpreted as the image source:

// Short syntax: String is src, then w h
Image "photo.jpg" 200 150

// With name (reusable)
Image Avatar "avatar.jpg" 48 48 rad 24

// Later simply:
Avatar

// Cover/Contain
Image "photo.jpg" 200 150 fit cover
Image "photo.jpg" 200 150 fit contain

// With alt text
Image "logo.png" alt "Company Logo"

Input

The string is interpreted as placeholder:

// Simple input
Input "Enter email..." w 200 pad 8 12 col #1E1E1E rad 6

// With name
Input EmailField "Email" w 200 pad 8 bor 1 boc #444

// Password
Input "Password" type password w 200

Textarea

Multi-line input:

Textarea "Write message..." w 300 h 150 pad 12

Textarea MessageField "Your message..." rows 5

Link

The string is interpreted as URL:

Link "https://example.com" col #3B82F6
  "Go to website"

Link HomeLink "https://home.com" col #3B82F6
  icon "home"
  "Home"

Context Menu

  • Opens on right-click on Trigger
  • Separator – Visual divider
  • Highlight destructive actions visually

Modifiers

Predefined style shortcuts. 

// Primary - Main action
Button -primary "Save"

// Secondary - Secondary action
Button -secondary "Cancel"

// Outlined - With border
Button -outlined "Details"

// Ghost - Transparent
Button -ghost "More"

// Disabled - Disabled
Button -disabled "Not available"

// Rounded - Pill shape
Button -primary -rounded "New"
Combining Modifiers
// Outlined + Rounded
Tag -outlined -rounded "React"

// Modifier + Override
Button -primary col #22C55E "Success"

Available Modifiers

  • -primary – Blue background color
  • -secondary – Gray background color
  • -outlined – Transparent with border
  • -ghost – Completely transparent
  • -disabled – 50% opacity, not clickable
  • -rounded – Pill shape (rad 99)
