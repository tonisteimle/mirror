Mirror

What is Mirror

Mirror is a description language for user interfaces. The code describes what should be visible. Mirror compiles to React components.

AI is changing how interfaces are built. Whether AI generates HTML or Mirror code – both work. The difference becomes apparent when reading and editing.

HTML/CSS

<button style="background-color: #2271c1;
  padding: 12px 24px; border: none;
  border-radius: 8px; color: white;
  font-family: system-ui;
  cursor: pointer;">
  Click me
</button>

Mirror

Button col #2271c1 pad 12 24 rad 8 "Click me"

Mirror describes what should be visible, not how it's technically implemented.

Components

All Elements are Components

Central definition is the default in Mirror: components for structures, tokens for values.

Every element is a component. The first usage defines it, subsequent ones inherit. 

Row gap 8
  Button col #2271c1, pad 8 16, rad 8, "Next"
  Button "Save"
  Button "Cancel"
  Button "Delete"

The first button sets background, padding, and radius. The following ones inherit this definition. Text color is automatically set based on background contrast.

Implicit versus Explicit Definition

The example above shows how you can define and display components directly while designing. Later, you might want to separate definition from usage. The only change needed is a colon: 

Button: col #2271c1, pad 8 16, rad 8

Button "Next"
Button "Save"
Button "Cancel"
Button "Delete"

The colon makes the difference: with a colon, the component is defined but not rendered. Without a colon, it is rendered. The definition can then be moved to a separate file if desired.

Hierarchical Structure

Interfaces are hierarchical: a card contains a title and text, a header contains a logo and navigation. In HTML, this means nested tags – opening tags, closing tags, and the actual content somewhere in between. In Mirror, indentation expresses this hierarchy directly. The visual structure of the code matches the visual structure of the UI. No closing tags, no tag soup – just clear nesting through whitespace. 

Footer pad 16 hor between
  Save-Area hor gap 8
    Button "Save"
    Button "Cancel"
  Navigation-Area hor gap 8
    Button "Back"
    Button "Next"
Save
Cancel
Back
Next

hor arranges children horizontally, between distributes available space between them. Indented elements are children of their parent.

Individual Instances

Each usage of a component can be customized:

The label becomes "Next" and the background changes to #5ba8f5.

But what about a footer with multiple buttons? If you write Footer Button #F00, which button turns red? Mirror can't know. The solution is to name the elements: 

Footer: pad 16 hor between
  Save-Area hor gap 8

    Cancel Button "Cancel"
  Navigation-Area hor gap 8
    Back Button "Back"
    Forward Button "Next"

    Button "Save"

Combining States and Inheritance

A base component can contain all possible elements and states. Variants inherit everything and override only what they need.

The Base Component

The base button defines three child elements: IconLeft, Label, and IconRight. Both icons are hidden by default. It also defines two states: hover and disabled. 

Button:
  hor gap 8 pad 12 24 rad 8 col #2271c1 col #FFF

  Label "Button"
  IconRight icon /chevron-right hidden

  state hover
    col #1a5a9e

  state disabled
    col #94A3B8 col #64748B opa 0.6

Variants

Each variant inherits all elements and states from the base. It only specifies what's different.

  Label "Next"

    col #2271c110

Tokens

Components centralize structure, tokens centralize values. A token is a named value that can be used throughout the design. Tokens start with $. 

$primary-col: #2271c1
$default-rad: 8
$default-pad: 16

Button $primary-col, $default-rad, $default-pad, "Save"
Link $primary-col, "Learn more"

Change $primary-col once, and every element using it updates. The suffix in the name (-col, -rad, -pad) tells Mirror which property it represents.

Try it

Change $primary to purple (#8B5CF6) and watch both buttons update.

Type Suffixes

The suffix tells Mirror the type. With a typed suffix, the property name can be omitted: 

// Without suffix – property required
$blue: #2271c1
Button col $blue

// With suffix – Mirror knows it's a color
$blue-col: #2271c1
Button $blue-col

Available suffixes:

Suffix Type Example
-col, -color Color $primary-col, $text-color
-size Font size $heading-size, $body-size
-pad, -spacing Spacing $card-pad, $section-spacing
-rad, -radius Border radius $button-rad, $card-radius

Semantic Token Hierarchy

Design systems often organize tokens in layers. Base tokens define raw values. Semantic tokens assign meaning. Component tokens connect to specific UI elements. 

// 1. Base colors – raw palette
$blue-500: #2271c1
$blue-600: #1a5a9e
$slate-100: #F1F5F9

// 2. Semantic tokens – meaning
$primary-col: $blue-500
$primary-hover-col: $blue-600
$surface-col: $slate-100

// 3. Component tokens – specific usage
$button-col: $primary-col
$button-hover-col: $primary-hover-col
$card-col: $surface-col

This structure allows changes at any level. Swap $blue-500 for a different shade, and all primary elements update. Reassign $primary-col to a different base color, and the meaning stays intact. Change $button-col independently if buttons need a unique treatment. 

Button $button-col
  state hover
    $button-hover-col

Interactivity

So far we've built static UIs. Now let's add behavior – events, actions, and state changes.

Traditional UI frameworks separate event handling into JavaScript callbacks, Redux actions, or hook functions. Mirror takes a different approach: behavior belongs where you see it. Events and actions are written directly on the component, making the connection between UI and behavior immediately visible.

Events

Events are written directly on the component. An event triggers an action. 

Button onclick toggle "Dark Mode"

onclick is the event. toggle is the action. The pattern is always: Component event action.

Available Events

  • onclick – when clicked
  • onchange – when value changes (inputs, selects)
  • oninput – while typing (real-time)
  • onfocus – when element gains focus
  • onblur – when element loses focus
  • onkeydown / onkeyup – keyboard events
  • onload – when component loads

Actions

Actions define what happens when an event fires. Mirror provides several built-in actions for common patterns.

Built-in Actions

  • toggle – switch between states
  • toggle ComponentName – toggle another component's visibility
  • open DialogName – open an overlay/dialog
  • close – close current overlay
  • show ComponentName – make component visible
  • hide ComponentName – hide component
  • page PageName – navigate to page
  • assign $var to value – set a variable
Examples
// Toggle own state
Button onclick toggle "Dark Mode"

// Open a dialog
Button onclick open Settings "Settings"

// Navigate to page
Link onclick page Dashboard "Go to Dashboard"

// Increment counter
Button onclick assign $count to $count + 1 "+"
Result
Dark Mode
Settings
Go to Dashboard
+

Why built-in actions? Instead of writing custom JavaScript handlers, Mirror provides semantic actions that describe intent. toggle means "switch states", open means "show an overlay". This keeps code readable and allows Mirror to handle the implementation details – state management, transitions, accessibility.

Try it

Add onclick toggle to make the button change between states when clicked.

States

Components can have multiple visual states. Each state defines different styling that applies when the state is active.

Events trigger actions, but where does the visual change happen? In Mirror, components can define states – named configurations of properties. When a state becomes active, its properties override the defaults. This separates "what triggers a change" (events) from "what the change looks like" (states).

In traditional development, you might toggle CSS classes with JavaScript, or manage state variables that control styling through conditionals. Both approaches scatter the visual definition across multiple places. With Mirror states, all visual variations live together on the component itself. You see at a glance what a button looks like when inactive, active, hovered, or disabled – without jumping between files or mental context-switching.

Defining States

Use the state keyword to define named states with their styling:

Mirror
Tab: pad 12 16 rad 8
  state inactive
    col transparent
  state active
    col #2271c1
Result (both states)
Inactive
Active

Use onclick toggle to switch between states:

System States

Some states are automatic – they're bound to browser pseudo-classes and don't need explicit event handlers. Mirror recognizes certain state names and automatically connects them to CSS pseudo-classes like :hover, :focus, and :active. Instead of writing onmouseenter/onmouseleave handlers to toggle hover styling, just define state hover and Mirror generates the appropriate CSS. This is both cleaner and more performant than JavaScript-based hover effects.

Automatic States

  • hover – when mouse hovers over element
  • focus – when element has keyboard focus
  • active – when being clicked/pressed
  • disabled – when element is disabled
Button: col #2271c1 pad 12 rad 8
  state hover
    col #1a5ba8    // darker blue on hover
  state active
    col #144d8c    // even darker when pressed
  state disabled
    col #333
    opacity 0.5

Named Instances

Give components a name to reference them later. Read and write their properties. Build interactive forms.

We've learned how events trigger actions. But how do we know what the user entered in a form? How do we change a button's text dynamically? Named instances solve this: give a component a name, and you can access its properties from anywhere.

Naming Components

Add a name after the component type to create a named instance: 

Input Email: placeholder "Enter email"
Input Password: placeholder "Enter password" type password
Button Submit: col #2271c1 "Login"

Now you can reference these components: Email.value, Password.value, Submit.disabled.

Component Properties

Every component has intrinsic properties you can read and write:

Reading
if Email.value
if Checkbox.checked
if Panel.visible
Writing
Submit.disabled = true
Error.visible = true
Error.text = "Invalid"

Common Properties

All components .visible, .disabled, .opacity, .col
Input / Textarea .value, .placeholder, .focus
Button .label, .loading
Checkbox / Switch .checked
Dialog / Overlay .open, .close
Text / Label .text
Image .src, .alt

Components ARE the state. No separate state management needed. Email.value is the email – simple as that.

Why this approach? Traditional frameworks require separate state variables (useState) that mirror UI values. In Mirror, the component is the source of truth. This eliminates the "state/UI sync problem" that plagues many applications.

Overlays

Dialogs, popovers, tooltips – UI elements that appear above the main content. Mirror provides built-in overlay components with automatic positioning and closing behavior.

Many interactions require showing content "on top" of the current UI – confirmation dialogs, dropdown menus, help tooltips. Instead of implementing z-index management, click-outside-to-close, and positioning logic yourself, Mirror provides overlay primitives that handle these behaviors automatically.

Dialog

A modal dialog that blocks interaction with the rest of the page. Define a trigger button and the dialog content: 

Button onclick open SettingsDialog "Open Settings"

Dialog SettingsDialog: col #1A1A23 pad 24 rad 12 w 400
  Title size 18 weight 600 "Settings"
  Text col #888 "Configure your preferences here."
  Actions hor gap 8 hor-r
    Button col #333 onclick close "Cancel"
    Button col #2271c1 onclick close "Save"

open SettingsDialog shows the dialog. close hides it. Dialogs automatically close when clicking outside or pressing Escape.

Positioning

Overlays can be positioned relative to their trigger element: 

// Position below the trigger
Button onclick open Menu below "Options"

// Position above
Button onclick open Tooltip above "Help"

// Center in viewport (default for dialogs)
Button onclick open Modal center "Open Modal"

Position Keywords

  • below – below the trigger (dropdowns)
  • above – above the trigger (tooltips)
  • left – to the left of trigger
  • right – to the right of trigger
  • center – centered in viewport (modals)

Dropdown

Animations

Elements can animate when appearing, disappearing, or continuously. Define entrance, exit, and looping animations.

Static UI feels lifeless. Animations communicate state changes, guide attention, and make interfaces feel responsive. But animation code is often verbose and scattered. Mirror centralizes animation definitions on the component itself – declare once, animate automatically whenever visibility changes.

Show & Hide Animations

Use show and hide blocks to define how an element animates in and out: 

Panel hidden
  show fade slide-up 300
  hide fade 200
  Content
    Text "Animated content"

The panel starts hidden (hidden). When it becomes visible, it fades in and slides up (300ms). When hiding, it fades out (200ms). Animations can be combined: fade slide-up.

Animation Types

Entrance/Exit
fade        // opacity
scale       // grow/shrink
slide-up    // from bottom
slide-down  // from top
slide-left  // from right
slide-right // from left
none        // instant
Continuous
spin   // rotation (loaders)
pulse  // opacity pulse
bounce // vertical bounce

Continuous Animations

Use the animate block for looping animations: 

Loader w 24 h 24
  animate spin 1000
  Icon icon loader

The icon rotates continuously, completing one full rotation every 1000ms.

Conditional Logic

Show or hide components based on conditions. Change properties dynamically. React to data.

Real interfaces adapt to context. A navigation changes when logged in. A button disables when a form is invalid. An error message appears only when needed. Mirror's conditional logic lets you express these rules declaratively – state your conditions, and the UI updates automatically.

Conditional Rendering

Use if and else to show different content based on a condition: 

if $isLoggedIn
  Avatar src $user.avatar
  Button onclick assign $isLoggedIn to false "Logout"
else
  Button onclick page Login "Sign In"

The condition can be any expression: a variable, a comparison, or a logical combination.

Conditional Properties

Change a single property based on a condition using inline if then else:

Block vs inline: Use if/else blocks when you want to show different components. Use inline if then else when the component stays the same but a property changes. 

Button if $isActive then col #2271c1 else col #333 "Status"

Badge if $count > 0 then visible else hidden $count

Text if $error then col #EF4444 else col #888 $message

Lists & Data

Render components for each item in a list. Access properties with dot notation. Bind data to the UI.

So far we've worked with static content. But real apps display dynamic data – lists of users, products, messages. Instead of manually repeating components, Mirror's each iterator generates UI from data arrays. Change the data, and the UI updates automatically.

The Each Iterator

Use each to create a component for every item in a list: 

each $task in $tasks
  Card col #1A1A23 pad 16 rad 8
    Title $task.title
    Status col #888 $task.status

The iterator variable ($task) is only available within the block. You can access nested properties with dot notation: $task.title, $task.user.name.

Data + Template
$users: [
  { name: "Alice", role: "Admin" },
  { name: "Bob", role: "Editor" },
  { name: "Carol", role: "Viewer" }
]

UserList ver gap 8
  each $user in $users
    Row hor between pad 12 col #1A1A23 rad 6
      Name $user.name
      Role col #888 $user.role
Result
AliceAdmin
BobEditor
CarolViewer

Nested Iteration

Iterators can be nested for hierarchical data: 

each $category in $categories
  Section ver gap 8
    Title size 16 weight 600 $category.name
    each $item in $category.items
      Item pad 8 $item.name

Data Binding

Variables are reactive. Display them directly, and the UI updates automatically when they change: 

// Display variable
Counter $count
Greeting "Hello, " $name
Price $amount " €"

// Change with assign
Button onclick assign $count to $count + 1 "+"
Button onclick assign $count to 0 "Reset"

The $event Object

In event handlers, $event contains information about what happened: 

// Get input value
Input onchange assign $name to $event.value

// Get checkbox state
Checkbox onchange assign $agreed to $event.checked

// Real-time input binding
Input oninput assign $search to $event.value

$event Properties

  • $event.value – current value (inputs, textareas)
  • $event.checked – checked state (checkboxes)
  • $event.target – the element that triggered the event

Centralized Events

For complex interactions, separate structure from behavior. Define all event handlers in one place.

Inline events work well for simple interactions. But as forms grow complex – with validation, error handling, and multi-step flows – scattering onclick handlers throughout your code becomes hard to maintain. The events block gathers all behavior in one place, creating a clear separation between what the UI looks like and how it behaves.

The Events Block

Use the events keyword to create a dedicated block for event handlers: 

// === COMPONENTS ===
Input Email: placeholder "Email"
Input Password: placeholder "Password" type password
Text Error: col #EF4444 visible false
Button Submit: col #2271c1 "Login"

// === LAYOUT ===
LoginForm: ver gap 16 pad 24 col #1A1A23 rad 12
  Input Email
  Input Password
  Text Error
  Button Submit

// === BEHAVIOR ===
events
  Email onchange
    Error.visible = false

  Submit onclick
    if Email.value and Password.value
      Submit.label = "Sending..."
      Submit.disabled = true
      page Dashboard
    else
      Error.text = "Please fill in all fields"
      Error.visible = true

Inline vs. Centralized

Use inline events for simple actions, centralized events for complex logic: 

// Inline – simple toggle
Button onclick toggle "Dark Mode"

// Centralized – validation, multiple actions
events
  Submit onclick
    if Email.value
      // validate, call API, navigate...

Structure and behavior separated. All interactions in one place. Reads like prose: "When Submit is clicked..."

Slots – Named Children

Components can define named positions for children. On usage, these slots are filled with content.

Good components are flexible. A Card should accept any header, any body, any actions. Slots let you define where content goes without prescribing what that content is. This creates truly reusable components that adapt to any use case.

Defining Slots

Child components with a colon become named slots: 

Card: ver pad 24 gap 16 col #1A1A23 rad 12
  Header: hor between ver-cen
  Title: size 20 weight 600
  Body: col #9CA3AF
  Actions: hor gap 8 hor-r

Filling Slots

When using the component, reference slot names to fill them: 

Card
  Header
    Badge col #10B981 "New"
    Icon icon more-vertical
  Title "Project Alpha"
  Body "A new project with great potential."
  Actions
    Button col #333 "Cancel"
    Button col #2271c1 "Save"

Flat Access

Access nested elements directly by name, no matter how deep they are: 

Header: hor between
  Left: hor gap 16
    Logo: w 100
    Nav: hor gap 8
  Right:
    Avatar: w 36 rad 18

// Usage – direct access to nested elements
Header
  Logo col #FF0000       // finds Logo in Left
  Avatar col #2271c1     // finds Avatar in Right

You don't need to repeat the hierarchy. Names are flat, structure is preserved. This makes components easy to use without knowing their internal structure.

Creating Multiple Items

Use - to create new instances instead of modifying existing ones: 

Menu: ver w 200 col #1A1A23 rad 8
  Item: pad 8 12

Menu
  - Item "Profile"
  - Item "Settings"
  - Item col #EF4444 "Logout"

Without - you modify an existing element. With - you create a new instance.

Time to make it look good. We only style the first occurrence of each element – all others inherit automatically through implicit inheritance.

The dashboard now has a dark theme. Notice how only the first Tile has colors and sizing defined – all four tiles look identical because they inherit from the first definition. Same for Value and Label inside.

Hard-coded colors make changes tedious. We extract design values into tokens. Using suffixes like -col and -rad, Mirror automatically knows which property to apply.

Now all colors are defined in one place. Change $card-col and all tiles update. The suffix -col tells Mirror to apply it as a color property – no need to write col $card-col.

Step 6: Separate Definitions

For larger projects, we separate tokens and component definitions from the actual content. The structure becomes clean and readable – just components and their content.

The content section now contains only structure and text – no styling details. All visual decisions live in the definitions above. In a real project, tokens and definitions could live in separate files, imported where needed.

This progression – from quick sketch to production-ready design system – happens naturally. Start messy, refine as you go. Mirror supports both rapid prototyping and clean architecture.
