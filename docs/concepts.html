Alles ist eine Komponente

Das fundamentale Prinzip: Jedes Element ist eine Komponente. Es gibt keinen Unterschied zwischen "primitiven Elementen" und "Komponenten". Komponenten können inline verwendet oder explizit definiert werden.

Inline – direkt verwenden
Button pad 12 col #3B82F6 "Click me"
Explizit definieren
Button: pad 12 col #3B82F6 rad 8

Der Doppelpunkt zeigt: Diese Zeile rendert kein UI. Sie definiert nur, wie Button aussehen soll.

Nach der Definition wiederverwendbar
Button: pad 12 col #3B82F6 rad 8

Button "Save"
Button "Cancel"
Button "Delete"

UI-Elemente sind immer wiederverwendbar – das führt automatisch zu Konsistenz. Es gibt keine Unterscheidung zwischen "Komponenten" und anderem Code. Wir denken immer in Komponenten, also schreiben wir auch so.

Einrückung = Struktur

Die Hierarchie entsteht durch Einrückung. Keine Tags, keine Klammern, keine Semikolons. 2 Spaces = ein Level tiefer. 

App
  Header
    Logo
    Nav
  Main
    Sidebar
    Content
      Article
  Footer
Mirror
Card
  Title "Hello"
  Body
    Text "World"
Äquivalent in HTML
<Card>
  <Title>Hello</Title>
  <Body>
    <Text>World</Text>
  </Body>
</Card>

Die visuelle Struktur entspricht der logischen – du siehst die Hierarchie direkt. Keine schließenden Tags, keine Klammern bedeutet weniger Syntaxfehler und etwa halb so viele Zeichen wie HTML.

Typinferenz

Werte werden automatisch interpretiert. Keine Typenangaben, keine Einheiten-Suffixe. Mirror erkennt was gemeint ist. 

// Zahlen → Pixel
pad 12          // → padding: 12px
w 200          // → width: 200px
gap 16         // → gap: 16px

// Hex → Farben (Unified Color System)
col #3B82F6     // → Container: backgroundColor, Text: color
col #1A1A1A     // → explizit backgroundColor
boc #33333380  // → border-color mit Alpha

// Strings → Text
"Hello World"  // → Textinhalt
font "Inter"   // → font-family: "Inter"

// Keywords → Boolean
hor            // → flex-direction: row
cen            // → zentriert
italic         // → font-style: italic

// Spezielle Werte
w full         // → width: 100%
h full         // → height: 100%

Du schreibst pad 12 statt padding: "12px". Keine vergessenen Einheiten, keine Anführungszeichen. Der Kontext bestimmt die Bedeutung – Mirror versteht, was gemeint ist.

Properties überschreiben

Jede Property kann bei der Verwendung überschrieben werden. Einfach die neue Property dranhängen – fertig.

Basis-Komponente
Button: pad 12 col #3B82F6 rad 8
Überschreiben bei Verwendung
// Andere Farbe
Button col #EF4444 "Delete"

// Größer
Button pad 20 size 18 "Large Button"

// Ghost-Stil
Button col transparent bor 1 boc #3B82F6 "Ghost"

Jede Komponente ist anpassbar – es gibt keine geschlossene Props-API. Einmalige Anpassungen brauchen keine vordefinierte Variante. Die Regel ist einfach: spätere Properties gewinnen.

Vererbung mit from

Erstelle Varianten einer Komponente, ohne alles zu wiederholen. Die neue Komponente erbt alle Properties und überschreibt nur was nötig ist. 

// Basis
Button: pad 12 col #3B82F6 rad 8

// Varianten
DangerButton from Button: col #EF4444
GhostButton from Button: col transparent bor 1 boc #3B82F6
SmallButton from Button: pad 8 size 12
RoundButton from Button: rad 999
Mehrfache Vererbung
SmallDangerButton from DangerButton: pad 8 size 12

Die Basis einmal definieren, Varianten davon ableiten. Änderungen an der Basis propagieren automatisch. Namen wie DangerButton kommunizieren Bedeutung, nicht nur Styling. Override für Einzelfälle, Vererbung für echte Varianten.

Referenzen mit $

Alles was mit $ beginnt ist eine Referenz. Ein einheitliches System für alle Arten von Werten.

Design Tokens
// Definition
$primary: #3B82F6
$spacing: 16

// Verwendung
Button col $primary pad $spacing
Komponenten-Properties
Card: rad 16 pad 20

// Referenz auf Card's radius
Button: rad Card.rad  // → 16
State-Variablen
// Variable anzeigen
Counter $count

// Variable ändern
Button onclick assign $count to $count + 1
Event-Objekte
Input onchange assign $name to $event.value
Checkbox onchange assign $checked to $event.checked
Iterator-Items
each $task in $tasks
  Card
    Title $task.title
    Status $task.status

Das $ signalisiert immer einen Verweis – egal ob Token, Variable, Event-Daten oder Iterator-Item. Alles funktioniert gleich, keine Imports nötig, alles automatisch verfügbar.

Layout

Flexbox-Layout mit minimaler Syntax. Richtung, Abstand, Ausrichtung – alles in wenigen Zeichen.

Richtung
// Horizontal (Zeile)
Nav hor gap 16
  Link "Home"
  Link "About"

// Vertikal (Spalte)
Sidebar ver gap 8
  Item "Dashboard"
  Item "Settings"
Ausrichtung
// Horizontal ausrichten
hor-l    // links
hor-cen  // zentriert
hor-r    // rechts

// Vertikal ausrichten
ver-t    // oben
ver-cen  // zentriert
ver-b    // unten

// Beides zentriert
cen
Praktisches Beispiel
Header: hor between ver-cen pad 16 col #1A1A1A
  Logo "Mirror"
  Nav hor gap 24
    Link "Docs"
    Link "Pricing"
  Button "Login"

Drei Zeichen statt zwanzig: hor ersetzt display: flex; flex-direction: row. Und hor-cen zentriert horizontal – egal ob das Layout horizontal oder vertikal ist. Kein Nachdenken über main und cross axis.

States – Visuelle Zustände

Komponenten können mehrere visuelle Zustände haben. Jeder State definiert, wie die Komponente in diesem Zustand aussieht. 

Toggle: w 52 h 28 rad 14 pad 2
  state off
    col #333
  state on
    col #3B82F6
  Knob: w 24 h 24 rad 12 col white
Tab-Komponente
Tab: pad 12 rad 8
  state inactive
    col transparent
  state active
    col #3B82F6

Alle Zustände einer Komponente sind an einem Ort sichtbar. Benannte States statt dutzender if-Abfragen. States sind ein Design-Konzept – so denken Designer bereits.

Events & Actions

Interaktivität durch Event-Handler. Ein Event löst eine Action aus.

Events
onclick    // Klick
onhover    // Hover
onchange   // Wert geändert
oninput    // Eingabe
onfocus    // Fokus erhalten
onblur     // Fokus verloren
Actions
toggle                    // State wechseln
change self to active   // Zu State
open Dialog              // Overlay öffnen
open Dialog slide-up    // Mit Animation
open Dialog fade 300   // Mit Animation + Dauer
close                     // Oberstes Overlay
close Dialog             // Spezifisches Overlay
page Dashboard           // Navigation
assign $x to 1            // Wert setzen
Animationen für open/close
slide-up slide-down slide-left slide-right fade scale none
Beispiele
Button onclick toggle "Dark Mode"
Button onclick open SettingsDialog slide-up 300 "Settings"
Button onclick close SettingsDialog fade 150 "Schließen"
Tab onclick change self to active "Tab 1"
NavItem onclick page Dashboard "Dashboard"
Button onclick assign $count to $count + 1 "+"

Du sagst was passieren soll, nicht wie. open Dialog slide-up – fertig. Keine Callbacks, keine Handler-Definitionen. Actions decken die allermeisten Fälle ab.

Benannte Instanzen

Gib Komponenten einen Namen, um sie später zu referenzieren. Wie in Visual Basic: Der Name ist die Identität.

Syntax
// Typ + Name = Benannte Instanz
Input Email: placeholder "Email" w full
Input Password: placeholder "Password" type password
Button Submit: pad 12 col #3B82F6 "Login"
Verwenden
LoginForm: ver gap 16
  Input Email
  Input Password
  Button Submit

Der Name macht die Komponente referenzierbar. Email, Password, Submit – du kannst sie später ansprechen.

Komponenten-Properties

Jede Komponente hat intrinsische Properties. Lesen und Schreiben mit Dot-Notation.

Property lesen
if Email.value                    // Truthy-Check
if Email.value == "test@test.com"  // Vergleich
if Checkbox.checked               // Boolean
if Panel.visible                  // Sichtbarkeit
Property schreiben
Submit.label = "Sending..."
Submit.disabled = true
Panel.visible = false
Error.text = "Bitte ausfüllen"
Image.src = "new-photo.jpg"

Komponenten SIND der State. Kein separates State-Management. Kein Redux. Kein Vuex. Die UI-Komponenten haben Properties – das ist der State.

Intrinsische Properties
// Alle Komponenten
.visible     // Sichtbar?
.disabled    // Deaktiviert?
.opacity     // Transparenz
.col         // Farbe (Hintergrund bei Containern)

// Input / Textarea
.value       // Aktueller Wert
.placeholder // Platzhalter
.focus       // Fokus setzen
// Button
.label       // Button-Text
.loading     // Lade-Zustand

// Checkbox / Switch
.checked     // Aktiviert?

// Dialog / Overlay
.open        // Offen?
.close       // Schliessen

// Text / Label
.text        // Textinhalt

Wie in Visual Basic: txtEmail.Text statt formState.email. Die Komponente ist ihr eigener State.

Zentrale Events

Trenne Struktur von Verhalten. Definiere alle Events an einem Ort.

Der events Block
events
  Email onchange
    Error.visible = false

  Submit onclick
    if Email.value and Password.value
      Submit.label = "Sending..."
      Submit.disabled = true
      page Dashboard
    else
      Error.text = "Bitte alle Felder ausfüllen"
      Error.visible = true
Vollständiges Beispiel
// === KOMPONENTEN ===
Input Email: placeholder "Email" w full
Input Password: placeholder "Password" type password
Text Error: col #EF4444 visible false
Button Submit: col #3B82F6 "Login"

// === LAYOUT ===
LoginForm: ver gap 16 pad 24
  Input Email
  Input Password
  Text Error
  Button Submit

// === VERHALTEN ===
events
  Submit onclick
    if Email.value and Password.value
      page Dashboard
    else
      Error.visible = true

Inline vs. Zentral

// Inline – für einfache Fälle
Button onclick toggle "Dark Mode"

// Zentral – für komplexe Logik
events
  Submit onclick
    // Validierung, API-Calls, Navigation...

Struktur und Verhalten getrennt. Alle Interaktionen an einem Ort. Lesbar wie Prosa: "Wenn Submit geklickt wird..."

Konditionale Logik

Zeige Komponenten basierend auf Bedingungen. Ändere Properties basierend auf State.

Bedingte Komponenten
if $isLoggedIn
  Avatar
  Button "Logout"
else
  Button "Login"
Bedingte Properties
// Einfach
Button if $isActive then col #3B82F6 else col #333

// Mehrere Properties
Card if $isSelected then col #3B82F6 bor 2 else col #1A1A1A bor 0

// Mit Vergleich
Badge if $count > 0 then opacity 1 else opacity 0

Die Syntax liest sich wie ein Satz: if $active then col blue. Keine kryptischen Ternary-Operatoren. Ändert sich der Wert, aktualisiert sich die UI automatisch.

Iteratoren

Rendere Komponenten für jedes Element einer Liste. Greife auf Eigenschaften des aktuellen Elements zu. 

each $task in $tasks
  TaskCard
    Title $task.title
    Status $task.status
    DueDate $task.due
Verschachtelte Iteration
each $category in $categories
  Section
    Title $category.name
    each $item in $category.items
      Item $item.name

Natürliche Sprache: each $task in $tasks. Die UI passt sich automatisch an die Daten an. Die Variable existiert nur im Block – keine Namenskonflikte möglich.

Slots – Benannte Kinder

Komponenten können benannte Positionen für Kinder definieren. Bei der Verwendung werden diese Slots gefüllt.

Definition mit Slots
Card: ver pad 24 gap 16 col #1E1E2E rad 12
  Header: hor between ver-cen
  Title: size 20 weight 600
  Body: size 14 col #9CA3AF
  Actions: hor gap 8 hor-r
Verwendung – Slots füllen
Card
  Header
    Badge "New"
    Icon "more-vertical"
  Title "Projektname"
  Body "Beschreibung des Projekts..."
  Actions
    Button "Abbrechen"
    Button col $primary "Speichern"

Die Definition legt fest, wo Inhalte erscheinen können. Die Verwendung bestimmt, was dort steht. Kinder werden direkt platziert, nicht als Props durchgereicht – wie bei Card.Title oder Card.Body in modernen UI-Bibliotheken.

Flacher Zugriff

Greife auf verschachtelte Elemente direkt per Namen zu. Egal wie tief sie in der Hierarchie liegen.

Verschachtelte Definition
Header: hor between h 60
  Left: hor gap 16
    Logo: w 100
    Nav: hor gap 8
  Right: hor
    Avatar: w 36 rad 18
Verwendung – direkter Zugriff
Header
  Logo col #FF0000       // Findet Logo in Left
  Avatar col #3B82F6     // Findet Avatar in Right

Du musst die Hierarchie nicht wiederholen. Namen sind flach, die Struktur bleibt erhalten. Das macht Komponenten einfach zu verwenden ohne ihre Interna zu kennen. Namen müssen eindeutig sein – bei Duplikaten gibt's einen Fehler.

List Items

Erstelle neue Instanzen mit -. Wie Listen in YAML.

Definition mit Vorlage
Menu: ver w 200 col #1E1E2E rad 8
  Item: pad 8 12 col #E4E4E7
Verwendung – mehrere Instanzen
Menu
  - Item "Profil"
  - Item "Einstellungen"
  - Item "Abmelden" col #EF4444
Kombination mit flachem Zugriff
Card: ver pad 16 gap 12
  Title: size 18
  Tags: hor gap 8
    Tag: pad 4 8 rad 4 col #3B82F6

Card
  Title "Projekt"         // Modifiziert
  - Tag "React"           // Neu
  - Tag "TypeScript"      // Neu

Ohne - modifizierst du ein existierendes Element. Mit - erstellst du eine neue Instanz. Einfache Unterscheidung, mächtige Flexibilität. Jedes Element kann so erweitert werden.

Datenbindung

Variablen sind reaktiv. Ändert sich ein Wert, aktualisiert sich die UI. Mit assign werden Werte geändert.

Variablen anzeigen
Label $username
Counter $count
Price $price " €"
Werte ändern mit assign
// Einfach setzen
Button onclick assign $count to 0 "Reset"

// Mit Expression
Button onclick assign $count to $count + 1 "+"
Button onclick assign $count to $count - 1 "-"

// Aus Input
Input onchange assign $name to $event.value
Expressions
// Arithmetik
Total $price * $quantity
Discount $total * 0.1

// In Bedingungen
if $count >= 10
  Badge "Maximum erreicht"

Zwei Wege, Werte zu setzen:
assign $variable to value – für Variablen (mit $ Prefix)
Component.property = value – für Komponenten-Properties (im events Block)

Ändert sich ein Wert, aktualisiert sich die UI – automatisch. Kein setState, keine Dispatches: einfach assign $count to $count + 1. Berechnungen wie $price * $quantity schreibst du direkt in die UI.

Content am Ende

Text-Inhalt steht immer am Ende der Zeile. Erst Komponente, dann Properties, dann Content. 

// Struktur
Komponente property value property value "Content"

// Beispiele
Button "Click"
Button col #3B82F6 "Click"
Button pad 12 col #3B82F6 rad 8 "Click"

// Dynamischer Content
Label $username
Greeting "Hallo, " $name
Price $amount " €"

Jede Zeile folgt demselben Muster und ist dadurch schnell lesbar. Links steht die Struktur, rechts der Inhalt. Immer dieselbe Reihenfolge: was es ist, wie es aussieht, was drin steht.

Geltungsbereich

Variablen haben unterschiedliche Sichtbarkeit. Von global bis lokal zum Block.

Global – Tokens
// Überall verfügbar
$primary: #3B82F6
$spacing: 16

// In jeder Komponente nutzbar
Button col $primary
Card pad $spacing
Global – State-Variablen
// Reaktive Variablen, überall sichtbar
Counter $count
Button onclick assign $count to $count + 1

// Andere Komponente kann dieselbe Variable lesen
Badge if $count > 0 then $count
Block-Scope – Iterator
each $task in $tasks
  // $task nur hier sichtbar
  Card $task.title

// Hier ist $task nicht mehr verfügbar
Footer
Handler-Scope – Event
Input onchange assign $name to $event.value
       // $event nur im Handler sichtbar

Du weißt immer, wo eine Variable gültig ist. $task in einem Iterator stört $task in einem anderen nicht. Design-Tokens sind global, Iterator-Items lokal – genau wie erwartet.

Auf einen Blick

// Komponente verwenden

  Body

// Slots

  Title:
  Body:

Ein einheitliches Modell: Komponente → Property → Wert. Alles andere baut darauf auf.

Tutorial

Ein Dashboard Schritt für Schritt bauen

Wir bauen ein komplettes Dashboard mit Sidebar, Header, Statistik-Karten, Datentabelle und Dialogen. Jeder Schritt zeigt neue Möglichkeiten.

Das Grundgerüst

Wir starten mit dem App-Layout: Sidebar links, Hauptbereich rechts.

Ziel

Ein zweispaltiges Layout mit fester Sidebar.

Code
App hor full
  Sidebar w 240 ver col #0F0F14
  Main grow ver

Die Regeln

  • hor – Kinder horizontal anordnen
  • full – 100% Breite und Höhe
  • w 240 – Feste Breite 240px
  • grow – Restlichen Platz einnehmen

Design Tokens definieren

Bevor wir weiterbauen, definieren wir unser Design-System. 

// Farben
$bg-app: #0A0A0F
$bg-sidebar: #0F0F14
$bg-card: #1A1A23
$bg-hover: #252530

$primary: #3B82F6
$success: #10B981
$warning: #F59E0B
$danger: #EF4444

$text: #F4F4F5
$text-muted: #71717A
$border: #27272A

// Abstände
$space-xs: 4
$space-sm: 8
$space-md: 16
$space-lg: 24
$space-xl: 32

// Radien
$radius-sm: 4
$radius-md: 8
$radius-lg: 12

Die Regel

Tokens sind globale Variablen. $name: wert definiert, $name verwendet.

Logo oben, Navigation in der Mitte, User unten.

Dynamische Listen

Projekte aus einer Datenliste rendern.

Daten
$projects: [
  { name: "Website Redesign", status: "active", progress: 75 },
  { name: "Mobile App", status: "active", progress: 40 },
  { name: "API Integration", status: "paused", progress: 20 },
  { name: "Dashboard", status: "done", progress: 100 }
]
Projekt-Karte
ProjectCard: hor between ver-cen pad $space-md col $bg-card rad $radius-md hover-col $bg-hover
  Info: ver gap $space-xs
    Name: weight 500
    Status: size 12 col $text-muted
  Progress: hor gap $space-sm ver-cen
    Bar: w 100 h 6 col $border rad 3
    Percent: size 12 col $text-muted w 36
Iteration
ProjectList ver gap $space-sm pad $space-lg
  each $project in $projects
    ProjectCard
      Info
        Name $project.name
        Status $project.status
      Progress
        Bar
        Percent $project.progress "%"

Die Regel

  • each $item in $list – Für jedes Element
  • $project.name – Auf Felder zugreifen
  • Iterator-Variable ist nur im Block sichtbar

Bedingte Styles

Status-Badges mit unterschiedlichen Farben je nach Wert. 

StatusBadge: pad $space-xs $space-sm rad $radius-sm size 12 uppercase
  if $status == "active"
    col #10B98120
  if $status == "paused"
    col #F59E0B20
  if $status == "done"
    col #3B82F620
Alternative: Inline Conditional
Status if $project.status == "active" then col $success else if $project.status == "paused" then col $warning else col $primary

Neue Konzepte

  • uppercase – Text in Großbuchstaben
  • #10B98120 – Farbe mit Alpha (20 = 12% Opacity)
  • Mehrere if Blöcke für verschiedene Werte

HTML Primitives

Bestimmte Elemente benötigen native HTML-Tags: Image, Input, Textarea, Link, Button. Diese werden mit einem Keyword markiert.

Syntax

Keyword Name? "Wert" Properties

Der Name ist optional und macht die Komponente wiederverwendbar.

Image

Der String wird als Bildquelle interpretiert:

// Kurzsyntax: String ist src, dann w h
Image "photo.jpg" 200 150

// Mit Name (wiederverwendbar)
Image Avatar "avatar.jpg" 48 48 rad 24

// Später einfach:
Avatar

// Cover/Contain
Image "photo.jpg" 200 150 fit cover
Image "photo.jpg" 200 150 fit contain

// Mit Alt-Text
Image "logo.png" alt "Company Logo"

Input

Der String wird als Placeholder interpretiert:

// Einfaches Input
Input "Email eingeben..." w 200 pad 8 12 col #1E1E1E rad 6

// Mit Name
Input EmailField "Email" w 200 pad 8 bor 1 boc #444

// Password
Input "Passwort" type password w 200

Textarea

Mehrzeilige Eingabe:

Textarea "Nachricht schreiben..." w 300 h 150 pad 12

Textarea MessageField "Deine Nachricht..." rows 5

Link

Der String wird als URL interpretiert:

Link "https://example.com" col #3B82F6
  "Zur Website"

Link HomeLink "https://home.com" col #3B82F6
  icon "home"
  "Startseite"

Fit-Optionen (Image)

  • fit cover – Füllt Container, schneidet ab
  • fit contain – Passt rein, zeigt alles
  • fit fill – Streckt/Staucht
