Interaktion

Wie Elemente auf Benutzer reagieren

Bisher haben wir statische UIs gebaut. Jetzt lernen wir, wie Elemente ihr Aussehen ändern können – bei Hover, bei Klick, oder wenn etwas anderes passiert. Das Konzept dahinter: States.

Das Konzept: States

Ein State beschreibt, wie ein Element in einem bestimmten Zustand aussieht. Ein Button kann zum Beispiel diese States haben:

  • Base – der Normalzustand (grauer Hintergrund)
  • hover – wenn die Maus darüber ist (hellerer Hintergrund)
  • on – wenn er aktiviert wurde (blauer Hintergrund)

In Mirror trennen wir klar:

on:State – definiert das Aussehen
onclick:Event – wann passiert etwas
toggle()Funktion – was passiert

Diese Trennung macht den Code lesbar: States beschreiben nur Styles, Events rufen Funktionen auf.

System-States: hover, focus, active

Manche States werden automatisch vom Browser ausgelöst – du musst keinen Trigger definieren. Der Browser weiß selbst, wann die Maus über einem Element ist:

Der hover: State definiert nur das Aussehen. Der Browser kümmert sich darum, wann dieser State aktiv wird (Maus drüber) und wann nicht (Maus weg).

System-States: hover: (Maus darüber), focus: (Tastatur-Fokus), active: (während Klick gedrückt). Diese brauchen keinen Event-Trigger.

Custom States und toggle()

Für eigene Interaktionen definierst du Custom States. Anders als System-States werden diese nicht automatisch ausgelöst – du brauchst einen Event-Trigger:

Hier passieren drei Dinge:

  1. on: definiert einen State namens "on" mit blauem Hintergrund
  2. onclick: sagt: "Bei Klick..."
  3. toggle() wechselt zwischen Base und dem State

Base ist der Normalzustand – die Styles, die du direkt am Element definierst (hier: bg #333). Du brauchst keinen extra "off" State.

Im State starten

Manchmal soll ein Element bereits im aktivierten State starten. Dafür gibst du den State-Namen bei der Instanz an:

Der zweite Button startet im on State (blauer Hintergrund). Ein Klick schaltet ihn auf Base zurück.

States können alles ändern

Bisher haben States nur Styles geändert (Farben, Größen). Aber States können auch komplett andere Kinder haben – wie Figma Variants:

Im Base-State zeigt der Button "Mehr zeigen" mit Pfeil nach unten. Im open-State wird alles ausgetauscht: anderer Text, anderes Icon.

Wie Figma Variants: Jeder State kann eine komplett andere Version der Komponente sein – nicht nur andere Farben, sondern andere Inhalte, andere Struktur.

Mehrere States

Was wenn du mehr als zwei Zustände brauchst? Ein Task kann "todo", "doing" oder "done" sein. toggle() erkennt das automatisch und cyclet durch alle States:

toggle() erkennt automatisch wie viele States du hast:

  • 1 State → binärer Wechsel: default ↔ state
  • 2+ States → Cycle: todo → doing → done → todo → ...
Wichtig: Der erste definierte State ist der Startzustand. Die Reihenfolge der Definition bestimmt die Cycle-Reihenfolge.

Nur einer aktiv: exclusive()

Bei Tabs oder Radio-Buttons soll immer nur ein Element aktiv sein. Wenn du eines aktivierst, werden alle anderen automatisch deaktiviert:

exclusive() macht zwei Dinge: Es aktiviert das geklickte Element und deaktiviert alle Geschwister des gleichen Typs.

Gruppierung: Mirror erkennt automatisch, welche Elemente zusammengehören – alle Tab-Instanzen mit dem gleichen Parent bilden eine Gruppe.

Auf andere Elemente reagieren

Manchmal soll ein Element sein Aussehen ändern, wenn ein anderes Element seinen State wechselt. Klassisches Beispiel: Ein Menü wird sichtbar, wenn ein Button aktiviert wird.

Dafür brauchst du zwei Dinge:

  1. Gib dem steuernden Element einen Namen mit name
  2. Referenziere diesen Namen mit Name.state:

MenuBtn.open: bedeutet: "Wenn das Element namens MenuBtn im State 'open' ist, wende diese Styles an." Das Menü startet hidden und wird sichtbar, sobald der Button aktiviert wird.

Praktisch: Accordion

Ein Accordion zeigt im geschlossenen Zustand nur einen Header, im offenen Zustand auch den Inhalt:

Der Base-State zeigt "Mehr anzeigen" mit Pfeil nach unten. Der open-State zeigt "Weniger anzeigen" mit Pfeil nach oben plus den zusätzlichen Inhalt.

Eigene JavaScript-Funktionen

Die eingebauten Funktionen (toggle, cycle, exclusive) reichen für Standard-Interaktionen. Für komplexere Logik – API-Calls, Validierung, mehrere Elemente steuern – schreibst du eigene Funktionen:

Btn: pad 12 24, bg #333, col white
  loading:
    bg #666
    "Wird gespeichert..."
  onclick: save()

async function save(element) {
  element.state = 'loading'    // State setzen
  await fetch('/api/save')        // API-Call
  element.state = 'default'    // zurück zum Normalzustand
}

Wichtig: Mirror übergibt automatisch das Element als ersten Parameter. Du hast sofort Zugriff auf element.state, element.toggle(), etc.

Eingebaute + eigene Logik kombinieren

Du kannst die eingebauten Funktionen auch innerhalb eigener Funktionen nutzen:

function handleClick(element) {
  element.toggle()           // eingebautes Toggle
  analytics.track('click')   // eigene Logik
  sidebar.state = 'open'     // andere Elemente steuern
}

Faustregel: Einfach → onclick: toggle(). Komplex → eigene Funktion.

Elemente per Name ansprechen

Mit name gibst du einem Element einen Namen. Dieser Name ist dann direkt in JavaScript verfügbar – ohne querySelector:

Button name saveBtn, "Speichern"
Text name statusLabel, "Bereit"

function updateStatus() {
  saveBtn.state = 'loading'
  statusLabel.text = 'Wird gespeichert...'
}

Verfügbare Properties

.stateAktuellen State lesen oder setzen
.textText-Inhalt lesen oder setzen
.valueInput-Wert lesen oder setzen
.visibleSichtbarkeit (true/false)
.bg, .colFarben setzen
.toggle()State wechseln
.cycle()Zum nächsten State

Weitere Events

Bisher haben wir nur onclick: verwendet. Aber es gibt viele andere Momente, auf die du reagieren kannst: Tastatureingaben, Fokus-Wechsel, Eingaben in Formularfeldern.

Hover: onhover

Während hover: ein System-State ist (CSS-Pseudoklasse), ist onhover: ein Event, das Funktionen aufruft. Der Unterschied: States definieren Aussehen, Events lösen Aktionen aus.

onhover: wird ausgelöst, wenn die Maus über das Element fährt. In diesem Beispiel wechselt die Card bei Hover den State – ähnlich wie hover:, aber mit explizitem Event.

Tipp: Für einfache Hover-Effekte (Farbe, Größe) nutze den hover: State. onhover: brauchst du, wenn du komplexere Logik benötigst oder mehrere States durchschalten willst.

Tastatur: onkeydown

Mit onkeydown: reagierst du auf Tastatureingaben. Du kannst spezifische Tasten angeben:

Klicke ins Suchfeld und drücke Enter – der Status wechselt. Hier wird toggle() verwendet, um den State zu demonstrieren.

Die wichtigsten Tasten:

onkeydown enter:Enter/Return-Taste
onkeydown escape:Escape-Taste
onkeydown space:Leertaste
onkeydown tab:Tab-Taste
onkeydown arrow-up:Pfeiltaste hoch
onkeydown arrow-down:Pfeiltaste runter
Wichtig: Keyboard-Events funktionieren nur auf fokussierbaren Elementen (Input, Button) oder Elementen mit focusable-Property.

Fokus: onfocus / onblur

onfocus: wird ausgelöst, wenn ein Element den Fokus erhält (z.B. Klick ins Input). onblur: wenn es den Fokus verliert.

Das Input zeigt beim Fokus einen blauen Rahmen (focus: System-State). Der Hilfetext erscheint durch State-Referenz auf EmailInput.focused.

Typische Anwendungsfälle:

  • onfocus: Hilfetext anzeigen, Validierung starten
  • onblur: Eingabe validieren, Wert speichern

Klick außerhalb: onclick-outside

Ein häufiges Pattern: Ein Dropdown oder Modal soll sich schließen, wenn der Benutzer woanders hinklickt. Dafür gibt es onclick-outside:

Das Dropdown öffnet sich bei Klick auf den Button. Klickst du irgendwo anders auf der Seite (nicht auf den Button), schließt es sich wieder durch onclick-outside: toggle().

Hinweis: onclick-outside: ignoriert Klicks auf das Element selbst und seine Kinder. Perfekt für Dropdowns, Modals und Popovers.

Formular-Events: oninput / onchange

Für Eingabefelder gibt es zwei wichtige Events:

  • oninput: – wird bei jedem Tastendruck ausgelöst
  • onchange: – wird ausgelöst, wenn das Feld den Fokus verliert (nach Abschluss der Eingabe)

Wann was?

oninput:Live-Suche, Zeichenzähler, Echtzeit-Validierung
onchange:Autosave, finale Validierung, API-Calls
