Eingebaute und eigene Funktionen

In Kapitel 6 hast du toggle() und exclusive() kennengelernt – zwei eingebaute Funktionen für State-Wechsel. Mirror bietet weitere eingebaute Funktionen für typische UI-Patterns. Wenn die nicht reichen, schreibst du eigene.

Syntax: Funktionen als Properties

Funktionen werden direkt als Properties geschrieben – genau wie bg oder pad. Der Unterschied: Sie lösen Aktionen aus statt Styles zu setzen.

Wenn du eine Funktion als Property schreibst, wird sie automatisch bei Klick ausgeführt. Das ist der häufigste Fall – deshalb ist Klick der Default.

SyntaxBedeutung
Button "X", toggle()Funktion bei Klick (Kurzform)
Button "X", show(Menu)Element zeigen bei Klick
Button "X", toggle(), show(Panel)Mehrere Funktionen

Faustregel: Kurzschreibweise für Klick-Events (99% der Fälle). Für andere Events wie Enter oder Escape gibt es Shorthands (onenter, onescape) – siehe States.

Eingebaute Funktionen: Übersicht

Mirror hat eingebaute Funktionen für die häufigsten UI-Patterns. Du musst sie nicht importieren – sie sind einfach da:

KategorieFunktionWas sie tut
Statetoggle()State wechseln (an/aus oder durch mehrere cyclen)
exclusive()Diesen aktivieren, alle Geschwister deaktivieren
Visibilityshow(Element)Element sichtbar machen
hide(Element)Element verstecken

Für Overlays (Dialoge, Dropdowns, Tooltips) gibt es spezialisierte Zag-Komponenten – siehe Overlays. Für Navigation (Tabs, SideNav) siehe Navigation.

Diese Funktionen decken 90% der typischen Interaktionen ab. Für komplexere Logik (API-Calls, Validierung, Berechnungen) schreibst du eigene Funktionen – dazu später mehr.

show() und hide(): Sichtbarkeit steuern

Mit show(Element) und hide(Element) machst du Elemente sichtbar oder versteckst sie. Das ist nützlich für:

  • Info-Boxen die bei Klick erscheinen
  • Menüs die sich öffnen und schließen
  • Hinweise die nach einer Aktion angezeigt werden

Was passiert hier?

  • name InfoBox gibt dem Element einen Namen, damit wir es referenzieren können
  • hidden macht das Element initial unsichtbar
  • show(InfoBox) macht es sichtbar wenn der Button geklickt wird
  • hide(InfoBox) versteckt es wieder

Wichtig: Das Ziel-Element braucht einen name. Ohne Namen kann die Funktion das Element nicht finden. Der Name muss eindeutig sein – wie eine ID.

Praktisch: Aufklappbares Menü

Ein klassisches Pattern: Ein Button öffnet ein Menü, ein "Schließen"-Button (oder Klick außerhalb) versteckt es wieder. So baust du das mit show() und hide():

Das Menü startet hidden. Der "Menü öffnen"-Button macht es sichtbar, der "Schließen"-Button versteckt es wieder. Einfach und klar.

Für echte Menüs: In Produktion würdest du die Menu-Komponente aus Overlays verwenden – die hat bereits Positionierung, Tastatur-Navigation und "Klick außerhalb schließt". Hier zeigen wir das manuelle Pattern zum Verständnis.

Eigene Funktionen

Die eingebauten Funktionen decken Standard-Patterns ab. Aber was wenn du mehr brauchst?

  • API-Calls – Daten vom Server laden oder speichern
  • Validierung – Formulareingaben prüfen
  • Berechnungen – Werte berechnen und anzeigen
  • Komplexe Logik – Mehrere Bedingungen, Schleifen, etc.

Dafür schreibst du eigene Funktionen. Mirror bietet dafür eine vereinfachte Syntax – wie Mirror selbst: Einrückung statt Klammern, kein Boilerplate.

Variablen: $ oder nicht?

In Funktionen unterscheidest du zwischen globalen Daten (mit $) und lokalen Variablen (ohne $):

KontextSyntaxBeispiel
Globale Daten$name$users, $tasks, $settings
Lokale Variablenametasks = ..., count = 0
Parameternamefunction foo(project)

$ markiert globale Daten – das sind die Daten aus deinen .data-Dateien (siehe Kapitel 8). Lokale Variablen und Parameter leben nur innerhalb der Funktion.

Beispiel: Button mit API-Call

Ein typisches Pattern: Ein Button löst einen API-Call aus, zeigt während des Ladens einen anderen State, und danach Erfolg oder Fehler:

// Mirror UI – Button mit States
SaveBtn as Button: pad 12 24, bg #333, col white, rad 6, save()
  loading:
    bg #666
    "Wird gespeichert..."
  success:
    bg #10b981
    "Gespeichert!"

SaveBtn "Speichern"

Der Button hat drei States: Base (grau, "Speichern"), loading (dunkelgrau, "Wird gespeichert..."), success (grün, "Gespeichert!"). Die Funktion save() steuert, welcher State wann aktiv ist:

// Funktion in vereinfachter Syntax
async function save(element)
  element.state = 'loading'
  response = await fetch('/api/save')

  if response.ok
    element.state = 'success'
  else
    element.state = 'base'

Was passiert hier?

  • element ist der Button, der die Funktion aufgerufen hat
  • element.state = 'loading' wechselt sofort zum Loading-State
  • await fetch(...) wartet auf die API-Antwort
  • Je nach Ergebnis wird success oder base gesetzt

JavaScript bleibt erlaubt

Du kannst auch normales JavaScript schreiben – die vereinfachte Syntax ist optional. Das ist nützlich wenn du bestehenden Code einbinden willst:

// Normales JavaScript funktioniert auch
async function save(element) {
  element.state = 'loading';
  const response = await fetch('/api/save');
  element.state = response.ok ? 'success' : 'base';
}

Faustregel: Eingebaute Funktion wenn möglich (toggle, show, hide). Eigene Funktion nur wenn nötig (API, Validierung, komplexe Logik). Vereinfachte Syntax für sauberen Code.

Beispiel: Statistik berechnen

Ein komplexeres Beispiel: Eine Funktion die Daten filtert, zählt und ein Ergebnis zurückgibt:

function BerechneStats(project)
  // Alle Tasks dieses Projekts holen
  tasks = $tasks where project == project
  total = tasks.count

  // Edge Case: Keine Tasks
  if total == 0
    return { done: 0, open: 0, progress: 0 }

  // Tasks nach Status filtern
  done = tasks where done == true
  open = tasks where done == false

  // Optional: Logging
  each task in open
    log(task.name + " ist noch offen")

  // Ergebnis zurückgeben
  return {
    done: done.count,
    open: open.count,
    progress: done.count / total
  }

Diese Funktion zeigt mehrere Konzepte:

  • $tasks where project == project – Query auf globale Daten
  • tasks.count – Aggregation
  • if / return – Early Return für Edge Cases
  • each task in open – Iteration
  • return { ... } – Objekt zurückgeben

Domänen-Methoden

Manche Funktionen gehören logisch zu einem Datentyp. Ein Projekt hat einen Gesamtaufwand, ein User hat eine Aktivitätshistorie. Diese Funktionen kannst du als Methoden der Collection definieren.

Methoden deklarieren

Am Anfang einer .data-Datei deklarierst du, welche Methoden die Collection hat:

// data/projects.data

// Methoden deklarieren (am Anfang!)
Gesamtaufwand()
Fortschritt()

// Dann die Daten
website:
  name: Website Relaunch
  status: active

app:
  name: Mobile App
  status: planning

Methoden definieren

Die Implementierung steht in einer separaten Funktions-Datei. Der Name beginnt mit collection.methodenname:

// Gesamtaufwand aller Tasks eines Projekts
function projects.Gesamtaufwand(project)
  tasks = $tasks where project == project
  return tasks.sum(aufwand)

// Fortschritt in Prozent (0-1)
function projects.Fortschritt(project)
  tasks = $tasks where project == project
  done = tasks where done == true

  if tasks.count == 0
    return 0

  return done.count / tasks.count

Methoden aufrufen

Im Mirror-UI rufst du die Methode auf einem konkreten Eintrag auf:

// Auf einem bestimmten Projekt
Text $projects.website.Gesamtaufwand() + "h"
Text ($projects.website.Fortschritt() * 100).toFixed(0) + "%"

// In einer Schleife
each project in $projects
  Text project.name + ": " + project.Gesamtaufwand() + "h"

Wann Domänen-Methode vs. allgemeine Funktion?

TypSyntaxWann verwenden
Domänen-Methodecollection.name()Operiert auf einem Datentyp (Projekt, User, Task)
Allgemeine Funktionname()Utility, Helper, Actions (save, validate, format)

Daumenregel: Wenn die Funktion "zu einem Ding gehört" (Aufwand eines Projekts, E-Mail eines Users), mach sie zur Methode. Wenn sie allgemein ist (speichern, validieren), mach sie zur Funktion.
