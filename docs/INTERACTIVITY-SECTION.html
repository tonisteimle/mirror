Interactivity

Everything so far has been static – components that look good but don't respond to users. Now we'll bring them to life.

Interactivity in Mirror follows one simple principle that explains everything:

EVENT → ACTION → STATE

Something happens → Mirror does something → The UI changes

That's the entire mental model. A user clicks a button (event), Mirror toggles it (action), and the button switches to its "on" state (state). Once you understand this flow, everything else is just variations and combinations.

Why states instead of variables? You could manage button appearance with variables and conditionals, but states keep all visual variations in one place. You see at a glance every possible appearance of your button – no jumping between files or hunting for where colors are defined.

Understanding States

We've been using states, but let's understand them properly. A state is a named collection of property overrides. When a state becomes active, its properties replace the defaults.

The base Card has bor 1 #333. When the "selected" state activates, it overrides the border to bor 2 #3B82F6 and adds a shadow. Properties not mentioned in the state (like pad and rad) stay the same.

System States

Some states are automatic – the browser activates them based on user interaction:

You don't need onclick or any action – the browser handles hover and active states automatically.

System States (Automatic)

  • hover – mouse is over the element
  • focus – element has keyboard focus
  • active – element is being clicked/pressed
  • disabled – element is disabled

Tabs: When Only One Can Be Active

The tabs above have a problem: you can activate multiple tabs at once. In a real tab bar, selecting one should deselect the others. This is called mutual exclusion.

Mirror provides two actions for this pattern:

  • activate self – set your own state to "active"
  • deactivate-siblings – set all siblings to "inactive"

Now only one tab can be active at a time. Click any tab – it activates, and its siblings deactivate.

Highlight vs Select: Menus and Lists

For menus and dropdowns, you need two different concepts:

  • Highlight – temporary visual feedback (hover, keyboard navigation)
  • Select – permanent choice (click to choose)

Think about a dropdown menu: as you move your mouse, items highlight. But they're not selected until you click.

Hover over items – they highlight (gray). Click one – it gets selected (blue). The highlight is temporary; the selection persists.

Selection Actions

  • highlight self – mark as highlighted (temporary)
  • highlight next / prev – highlight next/previous sibling
  • select self – mark as selected (permanent)
  • select highlighted – select whatever is currently highlighted
  • deselect self – remove selection
  • clear-selection – remove all selections

Navigation Targets

Actions like highlight and focus can target relative positions:

Relative Targets

  • self – the element that received the event
  • next – next sibling
  • prev – previous sibling
  • first – first in container
  • last – last in container
  • highlighted – currently highlighted element
  • selected – currently selected element

Timing: Debounce and Delay

Some interactions need timing control. Two common cases:

Debounce: Wait for User to Stop Typing

For search/autocomplete, you don't want to filter on every keystroke – wait until the user pauses:

debounce 300 waits 300ms after the last keystroke before executing the action. If the user keeps typing, the timer resets.

Delay: Wait Before Acting

Sometimes you need a brief pause before an action. Common use: keeping a dropdown open long enough to click it:

Without the delay, clicking a result would blur the input and hide the results before the click registers. The 150ms delay gives you time to click.

Timing Modifiers

  • debounce N – wait N ms after last event (resets on each event)
  • delay N – wait N ms before executing
