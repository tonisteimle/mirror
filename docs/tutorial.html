Mirror

What is Mirror

Mirror is a description language for user interfaces. One syntax for structure, styling, and interactions – that compiles to React.

Learn Mirror in 2 Minutes

The best way to learn Mirror is by doing. Every example below is interactive – edit the code, and watch the result update instantly. No setup, no compile step, just immediate feedback.

1. A Component

In Mirror, everything is a component. A component is simply a named piece of UI. Why names? Because names let you reuse things. Define once, use everywhere.

That's it. Button is the component name, "Hello World" is what it displays. No boilerplate, no ceremony – just what you want to see.

2. Add Styling

A plain button isn't very exciting. Let's add some style. Properties follow the component name as name value pairs, separated by commas.

Here, background sets the color, padding adds inner spacing, and radius rounds the corners. Notice the two forms: you can write background or just bg. Mirror lets you choose – full words for readability, shortcuts for speed. Both compile to the same result.

3. Reuse and Override

Here's where Mirror gets interesting. Once you style a component, it remembers. Use that component again, and it keeps all its properties. You only specify what's different – no repetition, no copy-paste.

Look at the second button – just Button "Save". It inherits the blue background, padding, and radius from the first button automatically. The third button overrides only the background to red. This is how Mirror eliminates redundancy while keeping your code readable.

4. Layouts and Children

Real interfaces aren't flat – they're nested. A card contains a title and a description. A sidebar contains menu items. In Mirror, indentation creates hierarchy. Children are indented under their parent.

The ver (vertical) property stacks children top-to-bottom. The g (gap) property adds consistent spacing between them. Label and Value are children of Tile – they're indented under it. This is how you build complex layouts from simple pieces.

5. Reusable Components

Now we're ready for the big idea: templates. Add a colon after a component name to define it as a reusable template. The template isn't rendered – it's a blueprint. You define the structure once, then use it many times, filling in different content.

See how powerful this is? Tile: defines the template with its structure (Label and Value) and styling. Then each Tile usage just fills in the content. Six dashboard tiles, but the styling code exists only once. Change the template, and all six update.

6. States

So far, our button looks the same whether you hover over it or not. But real interfaces respond to interaction. When you hover, something should happen – a color change, a subtle shift. In Mirror, you define these variations as states. The component keeps its default look, and the state block describes what changes.

The hover block kicks in when the mouse hovers over the button. The active block activates when you press it. These are system states – they activate automatically based on user interaction. You just describe what should look different.

7. Click Events

States change appearance, but what about behavior? When the user clicks, something should happen – navigate to a page, show a message, update a value. Events connect user actions to responses. The most common is onclick.

Simple: onclick followed by an action. Here, alert displays a message. But the real power comes when events control other elements – showing panels, hiding dialogs, switching views.

8. Toggle Visibility

Here's where it gets practical. You can make one element control another. Click a button, and a panel appears. Click again, and it disappears. The toggle action does exactly this – it flips between visible and hidden each time.

Notice how onclick toggle Details references Details by name. That's how elements talk to each other – by name. No IDs to hunt down, no refs to wire up. Just say which element you want to control.

9. Starting Hidden

Sometimes you want something to start invisible – a tooltip, a dropdown, an error message. The hidden property does exactly this. The element exists in your code, ready to appear, but users don't see it until you reveal it.

Message hidden starts invisible. When you click the button, toggle brings it into view. Click again, and it vanishes. This pattern is everywhere: FAQs, accordions, modals, tooltips.

That's Mirror in a nutshell: components, styling, inheritance, states, and events. The following sections explore each concept in depth – with more examples, more patterns, and more power.

Components in Components

You can also define each component separately at the top. This gives you a clear separation between structure (what things look like) and content (what goes where). The definitions at the top are your design system; the usage below is your layout.

Once defined, components combine freely. The Footer doesn't care what's inside it – Save-Area, Navigation-Area, or something completely different. This flexibility lets you reuse components in ways you didn't plan for.

Components don't enforce their children. A Footer with its between layout works just as well with two buttons directly as it does with nested areas. This loose coupling is intentional – it keeps your options open.

Slots – Flexible Components

Your Card component always has the same children. But what if you want different actions sometimes? Two buttons, three buttons, or an icon instead? Slots let you define where content goes without prescribing what that content is.

The solution is slots. Inside your component definition, add child components with colons. These become named placeholders – areas where content can go. The slot defines the styling; the usage fills in the content.

Here's a convenience: you can access nested elements by name directly, regardless of how deep they are. Logo is inside Left which is inside Header – but you just write Logo and Mirror finds it.

This flat access means you can rearrange a component's internal structure without breaking the code that uses it. Move Logo from Left to Right? The usage code stays the same. It's about intent: you want to style Logo, not navigate to it.

Tokens

Ein professionelles Design System basiert auf einem zweistufigen Token-System: Eine Basis-Palette mit reinen Farbwerten und semantische Tokens, die auf diese Palette verweisen.

Das Zwei-Stufen-System

Das Token-System in Mirror folgt dem gleichen Prinzip wie professionelle Design-Systeme (Tailwind, Radix, Figma Tokens): Eine Basis-Palette definiert reine Werte, semantische Tokens geben diesen Werten Bedeutung.

Warum zwei Stufen? Stell dir vor, du willst die Primärfarbe ändern. Mit einem einstufigen System müsstest du jeden Button, jeden Link, jedes Highlight einzeln anpassen. Mit dem Zwei-Stufen-System änderst du nur die Referenz – und alles passt sich an.

Jetzt kannst du die Primärfarbe global ändern, indem du nur $primary.bg auf einen anderen Palette-Wert zeigen lässt – z.B. $primary.bg: $green-500.

Basis-Palette

Die Basis-Palette enthält reine Farbwerte ohne semantische Bedeutung. Das Namensschema folgt dem Muster $farbe-stufe, wobei niedrigere Zahlen heller und höhere Zahlen dunkler sind.

Diese Palette ist deine Farbgrundlage. Sie sollte alle Farben enthalten, die du in deinem Projekt brauchst. Die Palette selbst hat keine Meinung darüber, was 'primär' oder 'Hintergrund' ist – das kommt erst in der nächsten Stufe.

Semantische Tokens (Bound Property Format)

Semantische Tokens geben den Palette-Werten Bedeutung. Das Besondere in Mirror: Der Property-Name ist Teil des Token-Namens. Das Format ist $name.property – zum Beispiel $primary.bg für den Primär-Hintergrund.

Die Property im Namen zeigt sofort, wofür der Token gedacht ist:

  • .bg → Background (Hintergrundfarbe)
  • .col → Color (Textfarbe/Vordergrund)
  • .pad → Padding (Innenabstand)
  • .gap → Gap (Abstand zwischen Elementen)
  • .rad → Radius (Eckenrundung)
  • .font.size → Font-Size (Schriftgröße, Compound Property)

Beachte die Namenskonvention: $elevated.bg ist 'eine Stufe erhöht' gegenüber $default.bg. Diese Hierarchie hilft dir, konsistente Tiefenwirkung zu erzeugen.

Spacing und Größen

Neben Farben definierst du auch Abstände, Radien und Schriftgrößen als Tokens. Das Namensschema verwendet Größenstufen: xs, sm, md, lg, xl.

Compound Properties wie $xs.font.size haben einen zusätzlichen Teil im Namen (.font), der die Kategorie angibt. Das ist nützlich für Properties, die mehrere Aspekte haben (font-size, font-weight, font-family).

Tokens in Komponenten verwenden

Mit dem zweistufigen System sehen deine Komponenten-Definitionen sauber und wartbar aus. Jeder Wert hat einen semantischen Namen, der seine Funktion erklärt.

Wenn du jetzt die Primärfarbe ändern willst, änderst du nur $primary.bg: $green-500 – und alle Buttons, Links und anderen primären Elemente passen sich automatisch an.

Komponenten-spezifische Tokens

Manchmal brauchst du Tokens, die nur für eine bestimmte Komponente gelten. Diese können auf semantische Tokens verweisen – eine dritte Ebene der Indirektion.

So kannst du später entscheiden: Sollen alle Buttons der Primärfarbe folgen ($button.bg: $primary.bg) oder eine eigene Farbe haben ($button.bg: $grey-700)? Die Änderung ist trivial.

Best Practices

Hier sind die wichtigsten Regeln für ein wartbares Token-System:

  • Palette zuerst: Definiere alle Basis-Farben, bevor du semantische Tokens erstellst
  • Semantik referenziert: Semantische Tokens sollten immer auf Palette-Werte verweisen, nie direkte Hex-Codes enthalten
  • Property im Namen: Verwende .bg, .col, .pad etc. um den Verwendungszweck klar zu machen
  • Konsistente Stufen: Nutze xs, sm, md, lg, xl für Größenabstufungen
  • Keine Magic Numbers: Jeder Wert sollte einen Token haben – keine hartcodierten Zahlen in Komponenten

Mit diesem System kannst du das gesamte Erscheinungsbild deiner App ändern, indem du nur die Token-Definitionen anpasst – keine einzige Komponente muss bearbeitet werden.

Component References

Neben Tokens kannst du auch Properties von anderen Komponenten referenzieren. Das ist nützlich, wenn du sicherstellen willst, dass zwei Komponenten immer den gleichen Wert haben.

Wann was verwenden? Tokens für allgemeine Design-Werte, Component References für Werte, die explizit mit einer anderen Komponente synchron bleiben sollen.

Understanding self

When you have multiple instances of a component (like three tabs), how do you make each one change only itself? That's where self comes in.

The problem: If you wrote change Tab to active, ALL Tab instances would change. But you want only the clicked tab to become active.

The solution: self refers to the exact instance that received the event – the specific element you just clicked, hovered, or interacted with.

Click any tab – only that specific tab becomes active. The key insight: self always means "me, the element you just interacted with."

You'll use self constantly with repeated components: tabs, list items, cards, buttons in a group. Whenever each instance needs to respond independently, self is the answer.

But wait – clicking one tab doesn't deactivate the others. We need something more...

Tabs: When Only One Can Be Active

The tabs above have a problem: you can activate multiple tabs at once. In a real tab bar, selecting one should deselect the others. This is called mutual exclusion.

Mirror provides two actions for this pattern:

  • activate self – set your own state to "active"
  • deactivate-siblings – set all siblings to "inactive"

Now only one tab can be active at a time. Click any tab – it activates, and its siblings deactivate.

Highlight vs Select: Menus and Lists

For menus and dropdowns, you need two different concepts:

  • Highlight – temporary visual feedback (hover, keyboard navigation)
  • Select – permanent choice (click to choose)

Think about a dropdown menu: as you move your mouse, items highlight. But they're not selected until you click.

Hover over items – they highlight (gray). Click one – it gets selected (blue). The highlight is temporary; the selection persists.

  • highlight self – mark as highlighted (temporary)
  • highlight next / prev – highlight next/previous sibling
  • select self – mark as selected (permanent)
  • select highlighted – select whatever is currently highlighted
  • deselect self – remove selection
  • clear-selection – remove all selections

Navigation Targets

Actions like highlight and focus can target relative positions within a container:

  • self – this specific instance (the one you clicked/hovered)
  • next – next sibling in the container
  • prev – previous sibling in the container
  • first – first element in the container
  • last – last element in the container
  • highlighted – the currently highlighted element
